## IP 位址

每台連上網路的裝置都有一個 IP 位址，作為裝置在網路上的唯一識別。資料封包依靠 IP 位址決定送達目標。

**IPv4** 格式：

```
93.184.216.34
```

四組數字，每組 0–255，以小數點分隔。IPv4 位址空間約 43 億個，已不敷全球裝置使用。

**IPv6** 格式：

```
2606:2800:220:1:248:1893:25c8:1946
```

十六進位加冒號，位址空間近乎無限。目前 IPv4 與 IPv6 並存，多數服務同時支援。

人類無法記憶 IP 位址，因此需要 Domain 與 DNS 將可讀名稱對應至 IP。

## Domain 與 DNS

Domain（網域）是人類可讀的名稱，如 `google.com`、`github.com`。DNS（Domain Name System）負責將 domain 解析為 IP 位址。

Domain 結構由右至左：

- **TLD（頂級網域）**：最右邊，像 `.com`、`.org`、`.tw`
- **二級網域**：你登記的名字，像 `google`、`github`
- **子網域**：最左邊，像 `www`、`mail`、`api`

例：`mail.google.com` — `.com` 是 TLD，`google` 是二級網域，`mail` 是子網域。

### DNS 解析流程

瀏覽器不直接持有 domain 對應的 IP，需經過多層查詢：

<DnsResolutionDiagram client:visible />

流程：瀏覽器 → Recursive Resolver（ISP 或如 Google `8.8.8.8`）→ Root Nameserver → TLD Nameserver → Authoritative Nameserver → 取得 IP 位址。

:::info
實際查詢通常僅需數毫秒，因為每一層皆有快取。查詢結果依 TTL 設定暫存，期間內不需重新查詢。
:::

變更 DNS 設定後，全球各地快取須逐步過期才會生效，此等待期稱為 **DNS propagation**，通常需數小時至 48 小時。

## TCP/IP

TCP/IP 是網路通訊的基礎協定組，將通訊分為四層：

<TcpIpLayersDiagram client:visible />

各層職責：

| 層級 | 負責內容 |
| :--- | :--- |
| **應用層** | HTTP、Email、DNS 等應用協定 |
| **傳輸層** | 資料分段、送達確認、流量控制 |
| **網路層** | 封包路由、IP 定址 |
| **網路存取層** | 實體訊號傳輸（電纜、Wi-Fi、光纖） |

### TCP vs UDP

傳輸層兩大協定：

| | TCP | UDP |
| :--- | :--- | :--- |
| 全名 | Transmission Control Protocol | User Datagram Protocol |
| 連線方式 | 三次握手建立連線 | 無連線 |
| 可靠性 | 確認送達，遺失重送 | 不保證送達 |
| 延遲 | 較高（確認開銷） | 極低 |
| 適用場景 | 網頁載入、檔案下載 | 線上遊戲、視訊通話 |

:::tip
TCP 三次握手流程：Client → SYN → Server → SYN-ACK → Client → ACK，三步驟確保雙方準備就緒後才開始傳輸。
:::

## HTTP 與 HTTPS

**HTTP（HyperText Transfer Protocol）** 定義瀏覽器與伺服器間的通訊格式：

- **Request**：方法（GET、POST 等）、路徑、Header
- **Response**：狀態碼（200 OK、404 Not Found 等）、Header、Body

拆開看實際文字格式——Request：

```http
POST /api/login HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 42

{"email":"aira@example.com","pw":"****"}
```

Response：

```http
HTTP/1.1 200 OK
Content-Type: application/json
Set-Cookie: sid=abc123; HttpOnly; Secure
Content-Length: 25

{"ok":true,"userId":42}
```

HTTP 為明文傳輸，中間節點可讀取所有內容。

**HTTPS** 在 HTTP 下加入 **TLS（Transport Layer Security）** 加密層。連線時雙方協商金鑰，後續傳輸皆以該金鑰加密，即使被攔截也無法解讀。

:::warning
現在幾乎所有網站都應該用 HTTPS。瀏覽器會對 HTTP 網站顯示「不安全」警告，搜尋引擎也偏好 HTTPS 網站。如果你在架站，讓 TLS 憑證這件事盡早處理好。
:::

TLS 憑證可以從 Let's Encrypt 免費取得，大部分的主機平台（Vercel、Netlify、Cloudflare）都會自動幫你設定好。

## CDN

你的伺服器可能放在美國，但你的使用者可能在台灣、日本、德國。物理距離是客觀存在的，光速是有限的，資料要跑 2000 公里，延遲就是比較高。

**CDN（Content Delivery Network）** 的解法是在全球各地部署快取節點（Edge Node）。使用者連線時，不是直接打去原始伺服器，而是打去離他最近的 Edge Node。

<CdnDiagram client:visible />

CDN 的核心概念是快取：

- **Cache Hit**：Edge Node 已經有這份內容的副本，直接回傳，原始伺服器完全不需要介入
- **Cache Miss**：Edge Node 沒有快取（第一次請求、或快取過期），才回去跟原始伺服器要，取得後再快取起來

靜態資源（圖片、CSS、JS、字型）特別適合 CDN，因為這些檔案不常變動，快取命中率高。

實際上你每天都在用 CDN，只是沒意識到：

- **jsDelivr**、**unpkg**：npm 套件的免費 CDN
- **Cloudflare**：除了 CDN，還提供 DDoS 防護、DNS 管理
- **AWS CloudFront**：AWS 生態系的 CDN，跟 S3 整合緊密

:::info
CDN 不只是速度的問題。當流量突然暴增，CDN 的快取節點可以吸收大量請求，讓原始伺服器不至於被打爆。這也是很多人在網站上線前優先設定 CDN 的原因。
:::

## 串起來看

把這些拼在一起，你輸入網址按下 Enter 之後發生的事，大概是這樣：

1. **DNS 查詢**：瀏覽器問 Recursive Resolver，最終拿到 IP 位址
2. **TCP 連線**：三次握手，與目標伺服器（或 CDN Edge Node）建立連線
3. **TLS 握手**：如果是 HTTPS，協商加密金鑰
4. **HTTP 請求**：瀏覽器發出 GET 請求，說明要哪個資源
5. **CDN 回應**：如果是 Cache Hit，Edge Node 直接回傳；如果是 Cache Miss，Edge Node 先跟原始伺服器拿，再回傳
6. **頁面渲染**：瀏覽器拿到 HTML，再繼續請求 CSS、JS、圖片，重複以上步驟

整個流程在 0.幾秒內跑完，每個環節都有相應的技術在背後支撐。理解這個流程不只是知識層面的好奇，也能幫助你在遇到效能問題、網路錯誤的時候，更快判斷問題出在哪個環節。
