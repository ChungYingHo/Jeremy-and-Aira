這篇是針對從沒用過 Claude 的人寫的**操作手冊**：每一節都標明「檔案放在電腦哪個路徑」、「打什麼指令」，照著做就能上手。

## 0. 先搞清楚 Claude 的幾種使用方式

| 類型 | 場景 | 開發者通常用這個 |
| :--- | :--- | :---: |
| **claude.ai**（網頁版） | 瀏覽器上對話，適合問問題、讀文件 | |
| **Claude Desktop**（桌面版，含 Cowork） | 桌面 app，有圖形介面，可以連本機檔案與 MCP | ✓ |
| **Claude Code**（CLI） | 跑在終端機裡的 AI agent，直接讀寫 codebase、跑指令 | ✓✓ |
| **API** | 自己寫程式呼叫 Anthropic API | 特定需求才用 |

這篇手冊主要講 **Claude Code**，因為它是開發工作流裡最核心的工具。Claude Desktop 的設定邏輯大致相通。

## 1. 安裝與啟動

### 1-1. 安裝 Claude Code

先確認有 Node.js 18 以上，然後裝全域套件：

```bash
npm install -g @anthropic-ai/claude-code
```

裝完後在任何目錄打：

```bash
claude --version
```

能印出版本號就代表裝好了。

### 1-2. 登入

第一次啟動會要求登入：

```bash
claude
```

照畫面指示開瀏覽器登入 Anthropic 帳號，完成後 token 會寫到 `~/.claude/` 底下，之後就不用每次登入。

### 1-3. 啟動 Claude Code

在**想處理的專案根目錄**啟動：

```bash
cd my-project
claude
```

啟動後進入互動介面，直接打自然語言就可以：

```
> 幫我看一下 src/utils/date.ts 這支檔案在做什麼
```

Claude 會讀檔、解釋，並在動手改之前先請求許可。

### 1-4. 下載 Claude Desktop

從 [claude.ai/download](https://claude.ai/download) 取得對應作業系統版本（Mac / Windows）。安裝完登入同一個 Anthropic 帳號，MCP 與 skill 設定可以和 Claude Code 共用。

:::info
Cowork 模式是 Claude Desktop 裡的「像 Claude Code 那樣可以讀寫本機檔案、跑指令」的模式。開啟 Cowork 後選一個工作資料夾，體驗上跟 Claude Code 類似，只是介面是圖形的。
:::

## 2. CLAUDE.md vs settings.json — 最重要的兩個設定

這兩個檔案定位完全不同，一句話總結：

> **`CLAUDE.md` 是「告訴 Claude 怎麼做事」，`settings.json` 是「告訴 Claude 能做什麼」。**

### 2-1. CLAUDE.md — 專案說明書

**用途**：告訴 Claude 這個專案的規範、架構、coding style、commit 規則、必要背景。每次啟動 Claude Code 自動讀入，之後產出的程式碼會遵守裡面的規則。

**檔案位置**：

| 層級 | 路徑 | 適用範圍 |
| :--- | :--- | :--- |
| 全域（個人偏好） | `~/.claude/CLAUDE.md` | 不管在哪個專案都套用 |
| 專案層級 | `<你的專案>/CLAUDE.md` | 只對這個專案生效 |
| 子資料夾層級 | `<你的專案>/<subdir>/CLAUDE.md` | 只在這個 subdir 工作時套用 |

Windows 的 `~` 會展開成 `C:\Users\<你的帳號>`，例如 `C:\Users\Jeremy\.claude\CLAUDE.md`。

**該放什麼**：

- 專案用的技術棧、目錄結構
- Coding style（命名規則、縮排、是否加分號）
- Commit message 規範、PR 流程
- 常見踩雷與 workaround
- 不要動哪些檔案

**範例 `CLAUDE.md`**：

````markdown
# my-blog 專案規範

## 技術棧
- Astro 5 + Svelte 5 + Tailwind 4

## Coding Style
- 單引號、不加分號、2 空格縮排
- 不用相對父層引用，用 `@/` alias
- 禁止 `var`，只用 `const` / `let`

## Commit 規則
- 格式：`<type>: <描述>`
- type：feat / fix / docs / refactor / chore
- 不要 amend 已 push 的 commit

## 絕對不要動
- `src/models/menu.ts`（路由 SSOT，改動要先討論）
- `.env`（含密鑰）
````

### 2-2. settings.json — 權限與工具設定

**用途**：控制 Claude Code 「能」做哪些事——允許哪些工具、哪些指令不用再問、要不要載入哪些 MCP server、hook 掛什麼。

**檔案位置**：

| 層級 | 路徑 | 優先順序 |
| :--- | :--- | :---: |
| 全域 | `~/.claude/settings.json` | 低 |
| 專案層級 | `<你的專案>/.claude/settings.json` | 中 |
| 專案本機覆寫（不進 git） | `<你的專案>/.claude/settings.local.json` | 高 |

`settings.local.json` 通常加到 `.gitignore`，放個人的 MCP token、個人偏好，避免 commit 到 repo。

**最常用的設定：permissions**

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test)",
      "Bash(git status)",
      "Bash(git diff)",
      "Read(**)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)"
    ]
  }
}
```

- `allow`：這些指令 Claude 執行前不用再問
- `deny`：這些永遠禁止，Claude 要跑會直接被擋下來

**其他常見設定**：

```json
{
  "model": "claude-opus-4-7",
  "env": {
    "ANTHROPIC_API_KEY": "..."
  },
  "mcpServers": { /* MCP 設定，見 §5 */ },
  "hooks": { /* Hook 設定，見 §4 */ }
}
```

### 2-3. 兩者的關係

```
你跟 Claude 說：「幫我修掉 lint error」

Claude 會：
  1. 先看 CLAUDE.md  → 知道這個專案用 ESLint、單引號、不加分號
  2. 再看 settings.json → 知道「npm run lint」已經被 allow，不用問
  3. 直接跑 npm run lint、分析結果、改 code
```

簡單記：**CLAUDE.md 是「規範」，settings.json 是「權限」**。兩個都可以放全域與專案兩層，專案層會覆蓋全域層。

## 3. Skill 系統

### 3-1. Skill 是什麼

**Skill** 是「預先封裝好、可重複使用的能力包」。每個 skill 有自己的描述、指令、相依的工具。Claude 在對話中判斷需要時會自動觸發，也可以主動呼叫。

官方預裝的幾個常見 skill：

| Skill | 功能 |
| :--- | :--- |
| `docx` | 讀 / 寫 / 編輯 Word 文件 |
| `pdf` | 讀 PDF、合併、拆分、加浮水印 |
| `pptx` | 產生 / 修改 PowerPoint 簡報 |
| `xlsx` | 處理 Excel，含公式與圖表 |
| `skill-creator` | 幫你生成新的 skill 模板 |

### 3-2. 查看 / 呼叫 skill

在 Claude Code 裡輸入：

```
/skills
```

會列出目前可用的 skill。要主動呼叫某個 skill：

```
/pdf 把 report-q1.pdf 和 report-q2.pdf 合併
```

### 3-3. Skill 檔案在電腦哪裡

| 層級 | 路徑 | 範圍 |
| :--- | :--- | :--- |
| 全域 | `~/.claude/skills/<skill-name>/` | 所有專案共用 |
| 專案層級 | `<你的專案>/.claude/skills/<skill-name>/` | 只對這個專案 |

每個 skill 是一個資料夾，裡面至少要有一個 `SKILL.md`：

```
~/.claude/skills/pr-reviewer/
├── SKILL.md          # skill 描述與使用說明
├── checklist.md      # 這個 skill 會參考的資料
└── scripts/
    └── run-review.sh # 可執行的 helper
```

`SKILL.md` 開頭用 frontmatter 寫 metadata：

```markdown
---
name: pr-reviewer
description: 根據團隊 code review checklist 產出審查報告
---

# PR Reviewer

## 使用方式
當使用者說「幫我 review 這個 PR」時，讀取 PR diff 並對照 checklist.md 檢查：
1. 命名一致性
2. Error handling
3. 測試覆蓋
...
```

### 3-4. 自己寫 skill

最快的方式是叫 Claude 幫你做：

```
/skill-creator 幫我建一個 skill：
功能是讀取 PR diff、對照我們的 code review checklist 產出審查報告
```

Claude 會在 `.claude/skills/` 下自動建好資料夾與 `SKILL.md`。

## 4. Hook 系統

### 4-1. Hook 是什麼

**Hook** 就是「特定事件發生時自動執行的 shell 指令」。可以拿來：每次 Claude 改完程式自動跑 lint、每次呼叫某個工具前自動備份、Claude 問權限時自動通知 Slack。

### 4-2. 有哪些事件點

| 事件 | 觸發時機 |
| :--- | :--- |
| `PreToolUse` | Claude 即將呼叫某個工具（Edit、Bash、Write…）**之前** |
| `PostToolUse` | 工具執行**完成之後** |
| `UserPromptSubmit` | 使用者送出訊息時 |
| `Notification` | Claude 顯示通知（例如等待使用者確認）時 |
| `Stop` | 整輪對話結束、Claude 停止輸出時 |
| `SubagentStop` | 子 agent 結束時 |

### 4-3. Hook 設定在哪

Hook 寫在 `settings.json` 的 `hooks` 欄位：

- 全域：`~/.claude/settings.json`
- 專案：`<你的專案>/.claude/settings.json`

### 4-4. 範例：改完檔案自動 lint

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --fix \"$CLAUDE_FILE_PATH\""
          }
        ]
      }
    ]
  }
}
```

說明：

- `matcher: "Edit|Write"`：只在 Claude 呼叫 Edit 或 Write 工具時觸發
- `command`：要執行的 shell 指令
- `$CLAUDE_FILE_PATH`：Claude 自動帶入被編輯的檔案路徑

### 4-5. 範例：commit 前先跑 test

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'git commit'; then npm test; fi"
          }
        ]
      }
    ]
  }
}
```

## 5. MCP（Model Context Protocol）

### 5-1. MCP 是什麼

**MCP** 是 Anthropic 推的開放協定，用來讓 Claude 跟**外部工具與服務**溝通。

Claude 本身不會「連 Notion」、「連 GitHub」——但只要那個服務有 MCP server，把它加到設定裡，Claude 就能直接操作。

常見的 MCP server：

| Server | 能做的事 |
| :--- | :--- |
| Notion | 搜尋、建立、編輯 Notion 頁面 |
| GitHub | 讀 PR / Issue、留言、建 PR |
| Linear | 管理 ticket |
| Figma | 讀取設計稿 |
| Slack | 發訊息、搜對話 |
| Filesystem | 本機檔案操作（更細粒度） |
| Chrome / Puppeteer | 操作瀏覽器、抓網頁 |

### 5-2. MCP 設定在哪

寫在 `settings.json` 的 `mcpServers` 欄位：

- 全域：`~/.claude/settings.json`
- 專案：`<你的專案>/.claude/settings.json`
- 本機私有（不進 git）：`<你的專案>/.claude/settings.local.json`

含 token 的設定建議放 `settings.local.json`，並把它加到 `.gitignore`。

### 5-3. 範例：接 Notion

```json
{
  "mcpServers": {
    "notion": {
      "command": "npx",
      "args": ["-y", "@notionhq/notion-mcp-server"],
      "env": {
        "NOTION_API_KEY": "secret_xxxxxxxxxx"
      }
    }
  }
}
```

設定完重新啟動 `claude`，就可以：

```
> 幫我搜 Notion 裡所有標籤是 "frontend" 的頁面，整理成清單
```

### 5-4. 範例：接本機 filesystem（細粒度權限）

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/jeremy/projects/my-blog"
      ]
    }
  }
}
```

這個 MCP server 只能存取指定資料夾，比 Claude Code 預設的全盤存取更安全。

### 5-5. 找 MCP server

- 官方 registry：[modelcontextprotocol.io/registry](https://modelcontextprotocol.io)
- 在 Claude 裡輸入 `search_mcp_registry` 搜關鍵字
- 大多數都可以 `npx` 直接跑，不用自己建置

## 6. 常用指令速查

在 Claude Code 互動介面內，以 `/` 開頭的是內建指令：

| 指令 | 功能 |
| :--- | :--- |
| `/help` | 顯示所有可用指令 |
| `/model` | 切換模型（`/model opus`、`/model sonnet`、`/model haiku`） |
| `/clear` | 清除當前對話、重新開始 |
| `/compact` | 壓縮目前對話（保留摘要、釋放 context window） |
| `/skills` | 列出可用的 skill |
| `/mcp` | 管理 / 檢視 MCP server 狀態 |
| `/config` | 編輯 settings.json |
| `/resume` | 接續上一段被壓縮或中斷的對話 |
| `/cost` | 查看當前 session 的 token 使用與費用 |
| `/exit` | 離開 Claude Code |

:::tip
`/compact` 跟 `/clear` 差別：`/compact` 會保留對話摘要，後續還能銜接；`/clear` 整個重開。長任務跑到 context 快滿時用 `/compact`。
:::

## 7. Model 選擇與 Prompt 技巧

### 7-1. 三個模型怎麼挑

| 模型 | 強項 | 適合的任務 |
| :--- | :--- | :--- |
| **Opus** | 最強的推理能力 | 架構設計、跨檔案重構、debug 複雜邏輯 |
| **Sonnet** | 速度與品質平衡 | 日常寫 code、code review、寫文件 |
| **Haiku** | 最快、最便宜 | 簡單格式轉換、批次檔案處理、快速查詢 |

:::tip
卡在難題想很久時切 Opus，寫 CRUD / 填 boilerplate 時切 Sonnet 或 Haiku。`/model` 可以在對話中即時切換，不用重開。
:::

### 7-2. Prompt：具體 vs 模糊

**模糊**（產出不可預期）：

```
幫我寫一個驗證函式
```

**具體**（產出貼近預期）：

```
幫我寫一個 TypeScript 函式 validateEmail：
- 參數：string
- 回傳：{ valid: boolean; error?: string }
- 規則：
  1. 不能為空
  2. 必須含 @ 和 .
  3. @ 前至少一個字元
  4. 不允許連續的 ..
- 位置：src/utils/validate.ts
- 風格：專案用單引號、無分號
```

差別在於提供了**輸入 / 輸出格式、規則、位置、風格**四個面向的限制。

### 7-3. 丟問題時帶上完整 context

Debug 時不要只說「build 壞了」，把完整 error 跟相關程式碼一起給：

```
我跑 npm run build 遇到這個錯誤：

<貼完整 stack trace>

相關檔案：

<貼 src/xxx.ts>
<貼 astro.config.mjs>

我預期的行為是 ...
```

## 8. 其他要知道的

### 8-1. Permission mode

Claude Code 啟動時可以切換權限模式：

```bash
claude --permission-mode plan        # 只規劃、不動手
claude --permission-mode acceptEdits # 自動允許改檔案
claude --permission-mode bypassPermissions # 全自動（危險）
```

預設是每次有副作用的操作都會問。新手建議維持預設。

### 8-2. 哪些檔案別 commit

個人用的 token、個人偏好設定放在這些路徑，記得加到 `.gitignore`：

```
.claude/settings.local.json
.claude/.env
```

團隊共用的設定放在 `.claude/settings.json` 並 commit，這樣大家拉下來就有一致的 hook / MCP 基礎設定。

### 8-3. 遇到 Claude 亂搞怎麼辦

- `Esc` 中斷目前輸出
- 「請停止」、「還原剛才的修改」——Claude 會嘗試回退
- 用 `git status` / `git diff` 確認實際改動
- 重要操作前自己先 `git commit`，有事可以 `git reset`

## 9. 總結

- **安裝**：`npm install -g @anthropic-ai/claude-code` → `claude`
- **規範 Claude 怎麼做事**：寫在 `CLAUDE.md`
- **控制 Claude 能做什麼**：寫在 `.claude/settings.json`
- **預裝能力**：用 `/skills` 看，放在 `~/.claude/skills/` 或 `.claude/skills/`
- **自動化事件**：`settings.json` 的 `hooks`
- **連外部服務**：`settings.json` 的 `mcpServers`
- **切模型 / 壓對話 / 算錢**：`/model`、`/compact`、`/cost`

完整生態可以整理成一張圖：

```
你 ─> Claude Code / Desktop
           │
           ├── 讀 CLAUDE.md     → 知道專案規範
           ├── 讀 settings.json → 知道能用哪些工具、hook、MCP
           │
           ├── 內建工具：Read / Edit / Write / Bash / Glob / Grep
           ├── Skill：可重複使用的任務模板（docx、pdf、自訂…）
           ├── Hook：事件驅動的 shell 指令
           └── MCP：外部世界（Notion / GitHub / Slack / Chrome…）
```

從這張圖出發，遇到「想讓 Claude 多做一件事」時，判斷流程是：

1. 是「改行為規範」嗎？→ 改 `CLAUDE.md`
2. 是「開放 / 禁止某個工具」嗎？→ 改 `settings.json` 的 `permissions`
3. 是「要自動跑某段腳本」嗎？→ 寫 `hooks`
4. 是「要連外部服務」嗎？→ 加 `mcpServers`
5. 是「重複使用的任務流程」嗎？→ 包成 `skill`
