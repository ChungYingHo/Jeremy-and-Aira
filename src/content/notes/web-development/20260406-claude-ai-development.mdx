## 為什麼是 Claude？

Claude 在程式開發場景的表現與其他 AI 工具有明顯差異：

- 處理長篇程式碼和複雜邏輯時較穩定，長對話中較少遺失上下文
- 較少產出不存在的 API 或套件（hallucination 較低）
- 傾向先理解需求再產出程式碼，而非直接輸出大量未必符合需求的 code

---

## Claude 的核心特色

- **超大 Context Window**：可一次輸入完整檔案甚至整個小專案，無需分批貼上
- **程式碼品質穩定**：產出的命名、結構、風格一致性高
- **理解能力強**：給定完整元件並指出有 bug 時，通常能精確定位問題點而非重寫整個元件
- **拒絕瞎猜**：較少捏造不存在的函式或套件，不確定時會明確表示

## Model 選擇與設定

Claude 有多個模型，適用於不同場景。

### 三種模型各司其職

| 模型 | 特色 | 適合場景 |
| :--- | :--- | :--- |
| **Opus** | 最強、最聰明，推理能力頂尖 | 複雜的架構設計、多檔案重構、需要深度理解的任務 |
| **Sonnet** | 速度和品質的平衡點 | 日常開發、code review、寫文件、一般問答 |
| **Haiku** | 最快、最便宜 | 簡單的格式轉換、快速查詢、批次處理 |

Claude Code 中以 `/model` 指令切換模型：

```bash
# 切換到 Opus（處理複雜任務）
/model opus

# 切到 Sonnet（日常使用）
/model sonnet

# 用 Haiku（快速簡單任務）
/model haiku
```

### System Prompt 設定

透過 API 或 Claude 網頁版的 Project 功能可設定 **System Prompt**，定義 Claude 的行為規範。

```
你是一個資深前端工程師，專精 Vue 3 和 TypeScript。
回答時請：
- 使用繁體中文
- 程式碼遵循 ESLint 標準（單引號、無分號）
- 解釋時搭配程式碼範例
```

Claude 在整個對話中會遵守這些規則。

在 Claude Code 中，對應機制為 `CLAUDE.md`，支援專案層級、使用者層級、資料夾層級的分層設定，Claude 啟動時全部讀入。

## Claude Code：直接在終端機裡協作

**Claude Code** 是 Anthropic 官方的 CLI 工具，屬於 AI agent 等級 — 可直接讀取 codebase、修改檔案、執行指令。

```bash
# 安裝
npm install -g @anthropic-ai/claude-code

# 在專案根目錄啟動
claude
```

啟動後以自然語言下達指令：

```
> 幫我把 src/utils/date.ts 裡的 formatDate 函式加上對 invalid date 的處理
```

Claude 會自動讀取檔案、理解現有程式碼、提出修改方案並等待確認。

### CLAUDE.md — 給 AI 的專案說明書

專案根目錄放置 `CLAUDE.md`，寫入專案規範、架構、注意事項。Claude 每次啟動時自動讀取，後續產出的程式碼會自動遵守這些規則。

範例：本部落格專案的 `CLAUDE.md` 定義了「單引號、不加分號、2 空格縮排」、「禁止相對父層引用」等規範。

### Worktree — 安全的平行開發

**Worktree** 功能會建立 git worktree，讓 Claude 在隔離的副本上工作，不影響當前分支。適用於正在 feature branch 開發時，需要 Claude 處理不相關任務的場景。完成後再決定是否合併。

```
> /worktree 幫我在新的 worktree 裡重構 utils 資料夾
```

### Permission 控制

Claude Code 在執行寫檔案、跑指令等操作前會請求許可。兩種模式：

- **Ask**（預設）：每次操作前詢問
- **Auto-allow**：對特定操作自動允許

在 `settings.json` 中可針對各工具分別設定，例如自動允許讀取但寫入需確認。

### Hooks — 自動化觸發

**Hooks** 在特定事件發生時自動執行 shell command。範例：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "npx eslint --fix $FILEPATH"
      }
    ]
  }
}
```

上述設定：每次 Claude 編輯或寫入檔案後自動執行 ESLint 修正，確保產出的程式碼符合 lint 規則。

## Cowork 模式 — 桌面版的協作體驗

**Claude Desktop** 的 **Cowork** 模式提供圖形介面版的 Claude Code 能力，可直接操作檔案系統、執行程式、管理任務。

Cowork 模式的功能：

- **讀寫檔案系統**：與 Claude Code 相同，附加圖形介面檔案預覽
- **執行終端機指令**：build、test、lint 等
- **管理任務清單**：自動拆解任務並追蹤進度，介面上可見即時狀態
- **即時預覽**：前端程式碼修改後可直接預覽結果

## Skill 系統

Claude 內建多種 **Skill**，每個 Skill 針對特定任務優化。

### 文件處理類

| Skill | 能做的事 |
| :--- | :--- |
| **docx** | 建立、讀取、編輯 Word 文件，支援表格、格式化、樣式 |
| **pdf** | 讀取 PDF、合併/拆分 PDF、加浮水印、旋轉頁面 |
| **pptx** | 製作簡報投影片，從零開始或是修改現有的 |
| **xlsx** | 處理 Excel 試算表，計算公式、製作圖表、格式化 |

使用範例：

```
> /pdf 把 report-q1.pdf 和 report-q2.pdf 合併成一個檔案
```

```
> /xlsx 讀取 sales-data.csv，幫我加一欄計算每月成長率，然後存成 .xlsx
```

### 設計相關類

UI/UX 相關 Skill：

- **design-critique**：從 usability、hierarchy、consistency 角度給出結構化回饋
- **accessibility-review**：WCAG 2.1 AA 無障礙審查，檢查顏色對比度、鍵盤導覽、觸控目標大小
- **design-system**：審查設計系統，找出命名不一致或 hardcoded 值
- **ux-copy**：撰寫或審查 UX 文案 — 按鈕文字、錯誤訊息、空狀態提示
- **design-handoff**：從設計稿生成開發交接文件，包含 layout、design tokens、component props
- **user-research**：規劃使用者研究，建立訪談大綱、可用性測試計畫
- **research-synthesis**：將訪談紀錄、問卷結果整理成主題和洞察

無障礙審查範例：

```
> /accessibility-review 幫我檢查這個頁面的 a11y
```

會檢查顏色對比、focus 狀態、alt text、ARIA 標籤等，列出不符合 WCAG 標準之處與修正建議。

### 自定義 Skill

透過 **skill-creator** 建立自訂 Skill：

```
> /skill-creator 建立一個 skill，功能是：
> 讀取 PR 的 diff，然後根據我們團隊的 code review checklist 產出審查報告
```

Claude 會封裝該 Skill 供重複使用。團隊中的重複性流程適合包裝為 Skill。

## MCP — 連接外部世界的橋樑

**MCP（Model Context Protocol）** 是 Anthropic 推出的開放協定，讓 Claude 與外部工具和服務溝通。只要對方提供 MCP 介面，Claude 即可直接互動。

### Notion 整合

設定 Notion MCP connector 後，Claude 可執行：

- **搜尋頁面**：依關鍵字搜尋 Notion 頁面
- **建立頁面**：在指定位置新增頁面
- **管理資料庫**：篩選、整理 Notion 資料庫項目

```
> 幫我搜尋 Notion 裡面所有標籤包含 "frontend" 的頁面，
> 然後整理成一個摘要清單
```

### 瀏覽器自動化

透過 Chrome MCP connector，Claude 可以直接操控瀏覽器：

- 開啟網頁、點擊元素、填寫表單
- 截圖當前頁面
- 讀取頁面上的文字內容
- 查看 console log 和 network requests

適用於 E2E 測試或網頁資料擷取：

```
> 幫我開啟 staging 環境的登入頁面，用測試帳號登入，
> 然後截圖首頁的 dashboard
```

### 其他連接器

MCP 為開放協定，社群持續開發新 connector。可用 `search_mcp_registry` 搜尋可用的 MCP server，或用 `suggest_connectors` 取得推薦。

常見 connector：
- **GitHub**：直接操作 PR、Issue、讀取 repo
- **Slack**：發送訊息、搜尋對話
- **Database**：直接查詢 PostgreSQL、MySQL 等
- **File System**：更細粒度的檔案操作

## 排程任務（Scheduled Tasks）

Claude 支援定時執行的排程任務，在背景自動執行。

### 用 Cron Expression 設定排程

```
> /schedule 每天早上 9 點，幫我檢查 GitHub 上有沒有新的 PR 需要 review，
> 如果有的話整理成清單通知我
```

以標準 cron expression 定義執行時間：

| 表達式 | 意思 |
| :--- | :--- |
| `0 9 * * *` | 每天早上 9 點 |
| `0 9 * * 1-5` | 每個工作日早上 9 點 |
| `*/30 * * * *` | 每 30 分鐘 |
| `0 18 * * 5` | 每週五下午 6 點 |

### 管理排程

```bash
# 列出所有排程
> /schedule list

# 更新排程
> /schedule update <task-id> --cron "0 10 * * *"

# 手動觸發一次
> /schedule run <task-id>
```

### 實用的排程場景

- **每日 Standup 摘要**：每天早上自動整理昨天的 git commit 和 PR 活動
- **週報生成**：每週五自動把這週的工作內容整理成週報格式
- **依賴檢查**：每週一檢查 `package.json` 裡面有沒有過時或有安全漏洞的套件
- **文件同步**：定時檢查 code 跟文件是不是有脫節的地方

## 實戰：有效使用 Claude

### 1. Prompt 品質決定產出品質

**模糊的 prompt：**

```
幫我寫一個驗證函式
```

**具體的 prompt：**

```
幫我寫一個 TypeScript 函式 validateEmail，接收 string 參數，
回傳 { valid: boolean; error?: string }。
規則：
1. 不能為空字串
2. 必須包含 @ 和至少一個 .
3. @ 前面至少要有一個字元
4. 不允許連續的 ..
```

具體 prompt 的產出結果：

```typescript
const validateEmail = (email: string): { valid: boolean; error?: string } => {
  if (!email.trim()) {
    return { valid: false, error: 'Email cannot be empty' }
  }

  if (!email.includes('@') || !email.includes('.')) {
    return { valid: false, error: 'Email must contain @ and .' }
  }

  const [local] = email.split('@')
  if (!local || local.length === 0) {
    return { valid: false, error: 'Invalid format before @' }
  }

  if (email.includes('..')) {
    return { valid: false, error: 'Consecutive dots are not allowed' }
  }

  return { valid: true }
}
```

### 2. 程式碼審查

Claude 可從效能、可讀性、潛在 bug、安全性等角度審查程式碼。

```
幫我 review 這段 code，注意：
- 有沒有潛在的 bug 或 edge case
- 效能上有沒有可以改善的地方
- 命名跟結構是否清晰

[貼上你的程式碼]
```

Claude 在 review 時擅長偵測：
- **未處理的 edge case**：空陣列、null、undefined
- **效能問題**：迴圈內重複計算、不必要的 re-render
- **型別問題**：TypeScript 型別過鬆或過緊

### 3. 除錯

將完整錯誤訊息與相關程式碼一起提供給 Claude，可快速定位問題。

```
我在跑 npm run build 的時候遇到這個錯誤：

[貼上完整的 error stack trace]

相關的程式碼在這裡：

[貼上相關檔案]
```

關鍵：提供完整的 error stack trace，而非僅描述「build 壞了」。

### 4. 寫文件和測試

```
幫這個函式寫 JSDoc，然後用 Vitest 寫單元測試，
至少涵蓋 happy path、edge case、error case：

[貼上函式]
```

產出的測試案例涵蓋範圍通常完整，`describe`/`it` 命名清晰。但仍需人工確認邏輯正確性 — AI 偶爾會產出表面通過但未測到核心邏輯的測試。

## 實際工作流範例

### 用 Claude 寫部落格文章

典型流程：

1. 給定主題方向、語氣、涵蓋範圍
2. Claude 產出初稿（架構、內容、程式碼範例）
3. Review 並給回饋（調整語氣、補充範例等）
4. 迭代修改至定稿

在 Claude Code 中可直接在 repo 建立 `.mdx` 檔案，自動遵守 `CLAUDE.md` 中定義的 frontmatter 格式與檔名規範。

### 用 Claude 做多輪程式碼審核

多輪掃描式 code review 流程 — 每一輪 Claude 執行：

1. 掃描整個 codebase 找出問題
2. 列出所有發現（bug、效能、安全性、可維護性）
3. 按嚴重程度排序
4. 逐一修復
5. 修完之後再掃一次，確認沒有引入新的問題

每一輪給定不同的 focus：

```
第一輪：找出所有可能的 runtime error 和未處理的 edge case
第二輪：檢查所有 API 呼叫的 error handling
第三輪：找出效能瓶頸和不必要的 re-render
第四輪：審查 TypeScript 型別定義是否夠嚴謹
...
```

### 用 Claude 做設計審核

透過設計相關 Skill 在實作前抓到設計問題：

```
> /design-critique 這是我們新的 Dashboard 設計稿 [附截圖]，
> 主要使用者是企業的 HR 主管，幫我檢查：
> 1. 資訊層級是否清晰
> 2. 操作流程是否直覺
> 3. 有沒有跟現有設計系統不一致的地方
```

會從 visual hierarchy、information architecture、interaction patterns 等維度給出結構化回饋。適合在設計 review 會議前先行審查。

### 用 Claude 管理 Notion

透過 MCP 整合操作 Notion：

```
> 幫我把今天 git log 裡面的所有 commit 整理成 Notion 的 Daily Log 頁面，
> 放在 Engineering > Daily Logs 底下，
> 格式是：commit hash、作者、commit message、修改的檔案數量
```

```
> 搜尋 Notion 裡面 Sprint 23 的所有 task，
> 把已完成的整理成本週的 Sprint Review 文件，
> 包含每個 task 的標題、負責人、完成日期
```

## 協作最佳實踐

### 給足上下文

提供給 Claude 的資訊應包含：

- 程式碼的用途與目的
- 預期達成的效果
- 限制條件（框架版本、coding style 等）

上下文越完整，產出品質越高。

### 分段處理複雜任務

大功能拆為小步驟，逐步確認後再推進：

```
第一步：先幫我定義資料結構和型別
第二步：寫 API 呼叫的邏輯
第三步：寫 UI 元件
第四步：串接資料和 UI
```

### 不要無腦複製貼上

Claude 產出的程式碼必須經過理解與審查後才能使用。常見風險：

1. 使用了專案未安裝的套件
2. 風格與專案不一致
3. 做出不符合業務邏輯的假設
4. 偶爾仍會產出有 bug 的程式碼

### 善用迭代

與 Claude 的協作是迭代過程：

1. 給初始需求，取得第一版
2. 指出需修改之處
3. Claude 修改後再次審查
4. 重複至滿意

## Claude 的限制

以下場景 Claude 幫助有限：

- **業務邏輯決策**：如欄位是否必填，需由 PM 或需求文件決定
- **效能瓶頸根因分析**：可提供優化建議，但實際瓶頸需用 profiler 量測
- **架構設計最終決策**：可列出方案 pros/cons，但拍板需由熟悉團隊與技術債歷史的人決定
- **機密程式碼**：避免將公司機密 code 貼到網頁版 AI；有隱私考量時使用 Claude Code 搭配適當設定

---

## 總結

Claude 的核心價值在於將重複性工作自動化，讓開發者專注於需要創意和判斷力的部分。生態系涵蓋 CLI（Claude Code）、桌面版（Cowork）、MCP 外部服務整合、Skill 系統、排程任務自動化，已超越單純的聊天機器人定位。
