## 為什麼是 Claude？

AI 工具百百種，ChatGPT、Gemini、Copilot... 每隔幾個月就有新東西冒出來。那為什麼這篇要特別拿 Claude 出來講？

其實很簡單，因為 Claude 在「寫程式」這件事上，表現真的很不一樣。

跟 ChatGPT 相比，Claude 在處理長篇程式碼和複雜邏輯時明顯更穩定。它不太會在長對話中「忘記」前面聊過的東西，也比較不會自己幻想出不存在的 API。更重要的是，它的回答傾向先理解你的需求再動手，而不是一股腦丟一坨 code 出來。

好啦，不是在幫 Claude 打廣告（雖然這篇確實是跟它合寫的），但用了一段時間之後，我確實覺得在開發工作上它是目前最順手的 AI 夥伴。

---

## Claude 的核心特色

在聊怎麼用之前，先快速整理幾個 Claude 跟其他 AI 工具比較不一樣的地方：

- **超大 Context Window**：Claude 的上下文視窗很大，可以一次餵進完整的檔案甚至整個小專案，不用像以前一樣把 code 切成好幾段分批貼
- **程式碼品質穩定**：它產出的 code 在命名、結構、風格上比較一致，不會上一段用 camelCase 下一段突然變 snake_case
- **理解能力強**：你把整個元件丟給它說「這邊有 bug」，它通常能準確找到問題點，而不是重寫整個元件
- **拒絕瞎猜**：它比較不會硬掰不存在的函式或套件，不確定的時候會跟你說它不確定

## Model 選擇與設定

Claude 不是只有一個模型，它其實有一整個家族，不同場景適合用不同的。

### 三種模型各司其職

| 模型 | 特色 | 適合場景 |
| :--- | :--- | :--- |
| **Opus** | 最強、最聰明，推理能力頂尖 | 複雜的架構設計、多檔案重構、需要深度理解的任務 |
| **Sonnet** | 速度和品質的平衡點 | 日常開發、code review、寫文件、一般問答 |
| **Haiku** | 最快、最便宜 | 簡單的格式轉換、快速查詢、批次處理 |

用白話講就是：Opus 是你請來的資深顧問，Sonnet 是你的日常 pair programming 夥伴，Haiku 是幫你跑腿的實習生。

在 Claude Code 裡面可以用 `/model` 指令切換模型：

```bash
# 切換到 Opus（處理複雜任務）
/model opus

# 切到 Sonnet（日常使用）
/model sonnet

# 用 Haiku（快速簡單任務）
/model haiku
```

### System Prompt 設定

如果你是透過 API 或 Claude 網頁版的 Project 功能在用，可以設定 **System Prompt**，也就是給 Claude 的「角色說明書」。

```
你是一個資深前端工程師，專精 Vue 3 和 TypeScript。
回答時請：
- 使用繁體中文
- 程式碼遵循 ESLint 標準（單引號、無分號）
- 解釋時搭配程式碼範例
```

這樣 Claude 在整個對話中都會遵守這些規則，不用每次都重複說明。

而在 Claude Code 裡面，這個角色就是交給 `CLAUDE.md` 來扮演。你可以在專案層級、使用者層級、甚至資料夾層級放不同的設定，Claude 會全部讀進去。

## Claude Code：直接在終端機裡協作

如果你只用過網頁版的 Claude，那你一定要試試 **Claude Code**。

Claude Code 是 Anthropic 官方的 CLI 工具，它不是那種只能問問題的聊天機器人，而是可以直接讀你的 codebase、改檔案、跑指令的 AI agent。講白了就是：你在 terminal 裡面開一個對話，Claude 可以直接看到你整個專案，然後幫你改 code。

```bash
# 安裝
npm install -g @anthropic-ai/claude-code

# 在專案根目錄啟動
claude
```

啟動之後，你可以直接用自然語言跟它溝通：

```
> 幫我把 src/utils/date.ts 裡的 formatDate 函式加上對 invalid date 的處理
```

它會自己去讀檔案、理解現有的程式碼、提出修改方案，然後問你要不要套用。整個流程就像在跟一個坐在你旁邊的工程師 pair programming。

### CLAUDE.md — 給 AI 的專案說明書

Claude Code 有一個很讚的設計：你可以在專案裡放一個 `CLAUDE.md`，裡面寫這個專案的規範、架構、注意事項。Claude 每次啟動都會先讀這個檔案，所以它一開始就知道你的 coding style、目錄結構、還有哪些地雷不能踩。

其實這篇文章所在的部落格專案就有用 `CLAUDE.md`，裡面寫了像是「單引號、不加分號、2 空格縮排」、「禁止相對父層引用」這些規範。所以 Claude 在幫忙寫 code 的時候，就會自動遵守這些規則，不需要每次都提醒它。

### Worktree — 安全的平行開發

Claude Code 有個蠻酷的功能叫 **worktree**。它會建立一個 git worktree，讓 Claude 在一個隔離的副本上工作，不會動到你正在開發的分支。

這在什麼時候有用？比如你正在 feature branch 上寫東西，突然想叫 Claude 幫你做另一個不相關的任務，但又不想讓它弄亂你目前的工作。Worktree 就是讓它在旁邊開一個獨立空間做事，做完之後你再決定要不要合併。

```
> /worktree 幫我在新的 worktree 裡重構 utils 資料夾
```

### Permission 控制

Claude Code 預設會在執行某些操作前問你要不要允許，像是寫檔案、跑指令等等。這個設計是為了安全，但有時候你信任它的操作，可以調整 permission 設定讓它跑得更順：

- **Ask**（預設）：每次都問你
- **Auto-allow**：對特定操作自動允許

你可以在 `settings.json` 裡面設定哪些工具自動允許，比如讓它自動讀檔案但寫檔案要問你。這樣既保持安全又不會一直被打斷。

### Hooks — 自動化觸發

Claude Code 支援 **Hooks**，就是在特定事件發生時自動執行的 shell command。比如：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "npx eslint --fix $FILEPATH"
      }
    ]
  }
}
```

上面這段設定的意思是：每次 Claude 編輯或寫入檔案之後，自動跑 ESLint 修正。這樣就能確保 Claude 產出的 code 一定符合你的 lint 規則。

## Cowork 模式 — 桌面版的協作體驗

除了 CLI 的 Claude Code，Anthropic 還有桌面版的 **Claude Desktop**，裡面有個很厲害的模式叫 **Cowork**。

Cowork 跟一般聊天的差別在於，它可以直接操作你電腦上的檔案、執行程式、管理任務。有點像是把 Claude Code 的能力搬到了圖形介面上，但又多了一些視覺化的功能。

在 Cowork 模式下，Claude 可以：

- **直接讀寫你的檔案系統**：跟 Claude Code 一樣，但有圖形介面的檔案預覽
- **執行終端機指令**：跑 build、test、lint 都行
- **管理任務清單**：它會自己拆解任務、追蹤進度，你可以在介面上看到目前做到哪裡
- **即時預覽**：改了前端程式碼之後，可以直接在旁邊看到結果

如果你不習慣在 terminal 裡面跟 AI 互動，Cowork 模式會是一個比較友善的入口。但說實話，用習慣 Claude Code 之後會覺得 CLI 的效率更高，因為少了滑鼠操作那些步驟。

## Skill 系統 — Claude 的十八般武藝

這個是很多人不知道的功能。Claude 不只會寫程式跟回答問題，它還內建了一堆 **Skill**，每個 Skill 都是針對特定任務優化過的能力。

### 文件處理類

你沒看錯，Claude 可以直接幫你生成和處理各種辦公室文件：

| Skill | 能做的事 |
| :--- | :--- |
| **docx** | 建立、讀取、編輯 Word 文件，支援表格、格式化、樣式 |
| **pdf** | 讀取 PDF、合併/拆分 PDF、加浮水印、旋轉頁面 |
| **pptx** | 製作簡報投影片，從零開始或是修改現有的 |
| **xlsx** | 處理 Excel 試算表，計算公式、製作圖表、格式化 |

舉個例子，你可以在 Claude Code 裡面直接說：

```
> /pdf 把 report-q1.pdf 和 report-q2.pdf 合併成一個檔案
```

或是：

```
> /xlsx 讀取 sales-data.csv，幫我加一欄計算每月成長率，然後存成 .xlsx
```

這些操作以前你可能要開 Python 寫個 script 用 openpyxl 之類的套件處理，現在一句話就搞定了。

### 設計相關類

如果你的工作會碰到 UI/UX，這些 Skill 非常實用：

- **design-critique**：丟一張設計稿或截圖給 Claude，它會從 usability、hierarchy、consistency 的角度給你結構化的回饋
- **accessibility-review**：跑 WCAG 2.1 AA 無障礙審查，檢查顏色對比度、鍵盤導覽、觸控目標大小等等
- **design-system**：審查你的設計系統，找出命名不一致或是 hardcoded 值
- **ux-copy**：幫你寫或審查 UX 文案 — 按鈕文字、錯誤訊息、空狀態提示
- **design-handoff**：從設計稿生成開發交接文件，包含 layout、design tokens、component props
- **user-research**：規劃使用者研究，建立訪談大綱、可用性測試計畫
- **research-synthesis**：把一堆訪談紀錄、問卷結果整理成主題和洞察

比如你剛做完一版設計，想知道有沒有無障礙的問題：

```
> /accessibility-review 幫我檢查這個頁面的 a11y
```

Claude 會針對顏色對比、focus 狀態、alt text、ARIA 標籤等一一檢查，然後列出不符合 WCAG 標準的地方和修正建議。

### 自定義 Skill

最酷的是你可以用 **skill-creator** 來建立自己的 Skill：

```
> /skill-creator 建立一個 skill，功能是：
> 讀取 PR 的 diff，然後根據我們團隊的 code review checklist 產出審查報告
```

Claude 會幫你把這個 Skill 封裝好，之後你就可以重複使用。如果你們團隊有某些重複性的流程，把它做成 Skill 可以省下大量時間。

## MCP — 連接外部世界的橋樑

**MCP（Model Context Protocol）** 是 Anthropic 推出的一個開放協定，讓 Claude 可以跟外部的工具和服務溝通。你可以把它理解成 Claude 的「插頭」— 只要對方有提供 MCP 介面，Claude 就能直接跟它互動。

### 這代表什麼？

以前你要用 AI 操作 Notion，大概得自己寫一個中間層，或是用什麼 Zapier 之類的自動化工具串。有了 MCP 之後，Claude 可以直接透過 MCP connector 去讀取和操作 Notion。

### Notion 整合

設定好 Notion 的 MCP connector 之後，Claude 就能：

- **搜尋頁面**：「幫我找 Notion 裡面關於 API 設計的文件」
- **建立頁面**：「在 Engineering Wiki 底下新增一頁，標題是 Vue 3 Migration Guide」
- **管理資料庫**：「把 Sprint Backlog 裡狀態是 Done 的項目整理成本週週報」

```
> 幫我搜尋 Notion 裡面所有標籤包含 "frontend" 的頁面，
> 然後整理成一個摘要清單
```

### 瀏覽器自動化

透過 Chrome MCP connector，Claude 可以直接操控瀏覽器：

- 開啟網頁、點擊元素、填寫表單
- 截圖當前頁面
- 讀取頁面上的文字內容
- 查看 console log 和 network requests

這在做 E2E 測試或者是需要從網頁上抓資料的時候特別好用。比如：

```
> 幫我開啟 staging 環境的登入頁面，用測試帳號登入，
> 然後截圖首頁的 dashboard
```

### 其他連接器

MCP 是開放協定，所以社群也在不斷地開發新的 connector。你可以用 `search_mcp_registry` 去搜尋有哪些可用的 MCP server，或用 `suggest_connectors` 讓 Claude 根據你的需求推薦適合的連接器。

常見的有：
- **GitHub**：直接操作 PR、Issue、讀取 repo
- **Slack**：發送訊息、搜尋對話
- **Database**：直接查詢 PostgreSQL、MySQL 等
- **File System**：更細粒度的檔案操作

## 排程任務 — 讓 Claude 定時幫你做事

Claude 支援 **Scheduled Tasks**，你可以設定定時執行的任務，讓它在背景自動幫你做事。

### 用 Cron Expression 設定排程

```
> /schedule 每天早上 9 點，幫我檢查 GitHub 上有沒有新的 PR 需要 review，
> 如果有的話整理成清單通知我
```

Claude 會幫你設定一個 cron job，時間到了就自動執行。你可以用標準的 cron expression 來定義時間：

| 表達式 | 意思 |
| :--- | :--- |
| `0 9 * * *` | 每天早上 9 點 |
| `0 9 * * 1-5` | 每個工作日早上 9 點 |
| `*/30 * * * *` | 每 30 分鐘 |
| `0 18 * * 5` | 每週五下午 6 點 |

### 管理排程

設定好之後，你可以隨時查看、修改或刪除排程：

```bash
# 列出所有排程
> /schedule list

# 更新排程
> /schedule update <task-id> --cron "0 10 * * *"

# 手動觸發一次
> /schedule run <task-id>
```

### 實用的排程場景

- **每日 Standup 摘要**：每天早上自動整理昨天的 git commit 和 PR 活動
- **週報生成**：每週五自動把這週的工作內容整理成週報格式
- **依賴檢查**：每週一檢查 `package.json` 裡面有沒有過時或有安全漏洞的套件
- **文件同步**：定時檢查 code 跟文件是不是有脫節的地方

## 實戰：怎麼有效地使用 Claude

好啦，功能介紹完了，來聊聊實際使用上的心得。

### 1. Prompt 是一切的基礎

跟 AI 溝通，prompt 寫得好不好差非常多。同樣是請 Claude 幫你寫一個函式，下面兩種問法的結果天差地遠：

**❌ 模糊的 prompt：**

```
幫我寫一個驗證函式
```

Claude 不知道你要驗證什麼，會產出一個很泛用但不符合你需求的東西。

**✅ 具體的 prompt：**

```
幫我寫一個 TypeScript 函式 validateEmail，接收 string 參數，
回傳 { valid: boolean; error?: string }。
規則：
1. 不能為空字串
2. 必須包含 @ 和至少一個 .
3. @ 前面至少要有一個字元
4. 不允許連續的 ..
```

這樣 Claude 就能精準地產出你要的東西：

```typescript
const validateEmail = (email: string): { valid: boolean; error?: string } => {
  if (!email.trim()) {
    return { valid: false, error: 'Email cannot be empty' }
  }

  if (!email.includes('@') || !email.includes('.')) {
    return { valid: false, error: 'Email must contain @ and .' }
  }

  const [local] = email.split('@')
  if (!local || local.length === 0) {
    return { valid: false, error: 'Invalid format before @' }
  }

  if (email.includes('..')) {
    return { valid: false, error: 'Consecutive dots are not allowed' }
  }

  return { valid: true }
}
```

### 2. 程式碼審查

這是我覺得 Claude 最好用的場景之一。你把一段 code 丟給它，請它幫你 review，它會從效能、可讀性、潛在 bug、安全性等角度給你建議。

```
幫我 review 這段 code，注意：
- 有沒有潛在的 bug 或 edge case
- 效能上有沒有可以改善的地方
- 命名跟結構是否清晰

[貼上你的程式碼]
```

我的經驗是，Claude 在 review 時特別擅長抓到：
- **未處理的 edge case**：比如空陣列、null、undefined
- **效能問題**：比如在迴圈裡重複計算、不必要的 re-render
- **型別問題**：TypeScript 的型別有沒有太鬆或太緊

### 3. 除錯

遇到 bug 的時候，直接把錯誤訊息和相關程式碼一起丟給 Claude，通常比你自己瞪著螢幕看半天有效率多了。

```
我在跑 npm run build 的時候遇到這個錯誤：

[貼上完整的 error stack trace]

相關的程式碼在這裡：

[貼上相關檔案]
```

重點是**把完整的 error message 給它**。很多人只會說「build 壞了」，但不給錯誤訊息，這就像跟醫生說「我不舒服」但不說哪裡不舒服一樣。

### 4. 寫文件和測試

寫文件跟寫測試，大概是大部分工程師最不想做的兩件事。Claude 在這方面幫了大忙。

```
幫這個函式寫 JSDoc，然後用 Vitest 寫單元測試，
至少涵蓋 happy path、edge case、error case：

[貼上函式]
```

它產的測試案例通常還蠻全面的，而且命名很清楚，`describe` 和 `it` 的描述一看就知道在測什麼。當然，產出來之後還是要自己看過確認邏輯正不正確，畢竟 AI 有時候會寫出「看起來對但其實沒測到重點」的測試。

## 實際工作流範例

講了那麼多功能，來看看實際日常工作中怎麼用。

### 用 Claude 寫部落格文章

其實你現在看到的這篇文章，就是跟 Claude 一起寫的。流程大概長這樣：

1. **我先給方向**：告訴 Claude 我想寫一篇關於什麼主題的文章，語氣要怎樣、要涵蓋哪些內容
2. **Claude 產出初稿**：它會根據我給的方向寫出第一版，包括架構、內容、程式碼範例
3. **我 review 然後給回饋**：「這段太官方了，口語一點」、「這邊要補充一個實際例子」
4. **迭代修改**：來回幾次之後，文章就定稿了

而且在 Claude Code 裡面做更方便，因為它可以直接在 repo 裡面建立 `.mdx` 檔案，連 frontmatter 格式、檔名規範都會自動遵守（因為 `CLAUDE.md` 裡有寫）。

### 用 Claude 做多輪程式碼審核

這個是我們實際在專案裡用過的做法。我們曾經對一個專案做了**十輪掃描式的 code review**，每一輪 Claude 會：

1. 掃描整個 codebase 找出問題
2. 列出所有發現（bug、效能、安全性、可維護性）
3. 按嚴重程度排序
4. 逐一修復
5. 修完之後再掃一次，確認沒有引入新的問題

十輪下來，專案品質有了很明顯的提升。這種深度的 review 如果是人工做，可能需要好幾天，但 Claude 幾個小時就跑完了。

重點是每一輪要給 Claude 不同的 focus：

```
第一輪：找出所有可能的 runtime error 和未處理的 edge case
第二輪：檢查所有 API 呼叫的 error handling
第三輪：找出效能瓶頸和不必要的 re-render
第四輪：審查 TypeScript 型別定義是否夠嚴謹
...
```

### 用 Claude 做設計審核

如果你負責前端，免不了要跟設計稿打交道。Claude 的設計相關 Skill 可以幫你在實作之前就抓到問題：

```
> /design-critique 這是我們新的 Dashboard 設計稿 [附截圖]，
> 主要使用者是企業的 HR 主管，幫我檢查：
> 1. 資訊層級是否清晰
> 2. 操作流程是否直覺
> 3. 有沒有跟現有設計系統不一致的地方
```

Claude 會給你結構化的回饋，從 visual hierarchy、information architecture、interaction patterns 等維度一一分析。這在設計 review 會議之前先跑一遍，可以提升討論的品質。

### 用 Claude 管理 Notion

如果你的團隊用 Notion 管理文件或專案，透過 MCP 整合，Claude 可以變成你的 Notion 助手：

```
> 幫我把今天 git log 裡面的所有 commit 整理成 Notion 的 Daily Log 頁面，
> 放在 Engineering > Daily Logs 底下，
> 格式是：commit hash、作者、commit message、修改的檔案數量
```

或者更進階的：

```
> 搜尋 Notion 裡面 Sprint 23 的所有 task，
> 把已完成的整理成本週的 Sprint Review 文件，
> 包含每個 task 的標題、負責人、完成日期
```

這種事情以前你可能要花半小時手動整理，現在一句話就搞定了。

## 協作的最佳實踐

用了一段時間之後，我整理出幾個跟 Claude 協作比較順的做法：

### 給足上下文

Claude 不是通靈的。你不能只說「把這個改好」，你得告訴它：

- 這段 code 的用途是什麼
- 你想達成什麼效果
- 有什麼限制條件（框架版本、coding style 等）

上下文給得越多，它的回答品質越高。

### 分段處理複雜任務

如果你要 Claude 幫你做一個大功能，不要一次把所有需求全部倒出來。拆成幾個小步驟，每一步確認沒問題再往下走。

```
第一步：先幫我定義資料結構和型別
第二步：寫 API 呼叫的邏輯
第三步：寫 UI 元件
第四步：串接資料和 UI
```

這樣不僅結果更好，你也更容易在過程中發現問題。

### 不要無腦複製貼上

這是最重要的一點。Claude 產出的 code **一定要看過、理解之後再用**。

原因很簡單：
1. 它可能用了你專案沒有安裝的套件
2. 它可能產出風格跟你專案不一致的 code
3. 它可能做了一些「看起來合理但不符合你業務邏輯」的假設
4. 偶爾它還是會產出有 bug 的 code

把 AI 當成一個很強的 Junior 工程師：它產出的東西需要你這個 Senior 去 review 跟把關。

### 善用迭代

不用期待第一次就拿到完美的結果。跟 Claude 的對話是一個迭代的過程：

1. 先給初始需求，拿到第一版
2. 看過之後指出要修改的地方
3. Claude 修改後再看一次
4. 重複直到滿意

這個過程其實就是一種 pair programming，只是你的 pair 是 AI。

## 什麼時候不該用 Claude

雖然 Claude 很好用，但有些情況它幫不了太多：

- **高度依賴業務邏輯的決策**：比如「這個欄位到底要不要必填」，這種事 AI 不會知道，你得去問 PM 或看需求文件
- **效能瓶頸的根因分析**：它可以給你優化建議，但真正的效能瓶頸需要你用 profiler 去量測
- **架構設計的最終決策**：Claude 可以給你好幾個方案的 pros/cons，但最後拍板的應該是你，因為你才了解整個團隊的情況和技術債的歷史
- **涉及機密的程式碼**：注意不要把公司機密的 code 直接貼到網頁版的 AI 上，如果有隱私考量，用 Claude Code 搭配適當的設定會安全許多

## 結語

AI 不會取代工程師，但會用 AI 的工程師會取代不會用的。

Claude 對我來說，最大的價值不是「幫我寫 code」，而是「讓我可以把時間花在更重要的事情上」。把重複性的工作交給它，把需要創意和判斷力的部分留給自己，這才是人跟 AI 協作的正確姿勢。

而且 Claude 的生態系真的越來越完整了 — 從 CLI 的 Claude Code、桌面版的 Cowork、到 MCP 連接各種外部服務、Skill 系統處理各種文件格式、排程任務自動化日常工作... 它已經不只是一個「聊天機器人」了，它更像是你工作環境裡的一個全能助手。

好啦，如果你還沒試過 Claude，真的可以去玩玩看。不管是網頁版、Claude Code 還是桌面版，先用用看，跟它聊幾次，你就會慢慢找到你自己的使用節奏。
