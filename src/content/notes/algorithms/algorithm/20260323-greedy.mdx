## 貪婪是一種策略，不是一個演算法

貪婪（Greedy）是一種解題的**策略分類**，不是特定演算法。Prim、Kruskal、Dijkstra、Huffman 都屬於貪婪演算法，但解的問題各不相同。

共同點：**每一步都選當下最佳選項，不回頭。**

---

以找零問題為例：付 100 元買 73 元的東西，需找 27 元。貪婪策略：

1. 還要找 27 元，最大面額不超過 27 的硬幣是 10 元 → 拿一枚，剩 17 元
2. 還要找 17 元，最大面額不超過 17 的硬幣是 10 元 → 拿一枚，剩 7 元
3. 還要找 7 元，最大面額不超過 7 的硬幣是 5 元 → 拿一枚，剩 2 元
4. 還要找 2 元，拿兩枚 1 元

結果：10 + 10 + 5 + 1 + 1 = 27 元，5 枚，即為最少枚數。

:::warning
台灣面額（1、5、10、50、100、500）恰好適合貪婪。換一套面額則不一定——

面額 **1、3、4 元**，找 **6 元**：貪婪選 4+1+1 = **3 枚**，但 3+3 = **2 枚** 才是最優解。

任意面額的找零問題需用動態規劃才能保證正確。
:::

## 貪婪演算法與最佳化問題

多數貪婪演算法在解**最佳化問題（Optimization Problem）**——從可行解中找出最小成本、最大利潤或最少步驟的解。

最佳化問題的結構：

1. **候選集合**：所有可能拿來用的元素（邊、物品、字元⋯）
2. **可行解**：滿足問題限制條件的一組元素
3. **目標函數**：用來衡量一個可行解有多好（總成本、路徑長度、編碼長度⋯）
4. **最優解**：讓目標函數達到最大（或最小）的可行解

貪婪做法：每步從候選集合挑出「目前最好」的元素加入，不再更改。若問題結構正確，所得即為最優解。

### 貪婪成立需要兩個條件

貪婪能得到正確答案，需問題本身具備兩個特性：

**貪婪選擇性質（Greedy Choice Property）**

每步的貪婪選擇不會讓之後的最優解消失——局部最優能組合成全域最優。

**最優子結構（Optimal Substructure）**

整體最優解包含子問題的最優解。

兩個條件缺一不可。若問題同時具備，貪婪通常能在低複雜度下得到正確答案。

:::info
判斷能不能用貪婪，通常需要用反證法或交換論證來數學證明。實務上最快的做法是：先試著想貪婪策略，再找反例——能找到反例就換動態規劃。
:::

### 什麼時候適合用貪婪

判斷流程：

1. **尋找排序策略**：活動選擇按結束時間、Kruskal 按邊權重、Dijkstra 按距離
2. **驗證正確性**：用交換論證說明「換成貪婪的選擇不會更差」
3. **找反例**：若能找到讓貪婪失效的輸入，改用動態規劃

經典適用場景：最小生成樹、單源最短路徑、Huffman 編碼、活動選擇、分數背包。

## 最小生成樹

:::info
Prim、Kruskal、Sollin、Dijkstra 都以圖作為輸入。無向圖、有向圖、加權圖、路徑、樹等基本術語與圖解，請見《[Data Structure] 圖》與《[Data Structure] 樹》。以下複雜度分析中，$|V|$ 代表頂點數，$|E|$ 代表邊數。
:::

**最小生成樹（Minimum Spanning Tree，MST）** 解的問題是：在一個連通加權無向圖裡，找出一棵包含所有節點、且邊的權重總和最小的樹。

現實例子：幾個城市要連接電纜，每對城市之間鋪設的成本不同。希望所有城市都能互通，但總成本最低——這就是 MST 問題。

解 MST 有三個經典的貪婪演算法：Prim、Kruskal 和 Sollin，策略不同但結果相同。

以下三節共用同一張圖來走範例：

- 節點：A、B、C、D、E、F（共 6 個）
- 邊（含權重）：A-B(4), B-C(3), D-E(7), E-F(9), A-D(1), B-E(2), C-F(6), A-E(8), C-E(5)

**MST 最終結果**：$\{\text{A-D}(1),\ \text{B-E}(2),\ \text{B-C}(3),\ \text{A-B}(4),\ \text{C-F}(6)\}$，總權重 **16**。三種演算法挑邊順序不同，但結果一致。

### Prim 演算法

**策略：從任意節點出發，每次把「連到一個新節點的最便宜邊」加進樹，直到所有節點都在裡面。**

概念上像一棵樹不斷「長大」，每步吸收最近的新節點。

#### 手把手：從 A 出發走一遍

| 輪 | 目前樹內 | 候選邊（權重） | 選擇 | 總權重 |
| :---: | :--- | :--- | :---: | :---: |
| 1 | \{A\} | A-D(1), A-B(4), A-E(8) | **A-D(1)** | 1 |
| 2 | \{A, D\} | A-B(4), D-E(7), A-E(8) | **A-B(4)** | 5 |
| 3 | \{A, D, B\} | B-E(2), B-C(3), D-E(7), A-E(8) | **B-E(2)** | 7 |
| 4 | \{A, D, B, E\} | B-C(3), C-E(5 跳\*), D-E(7 跳\*), A-E(8 跳\*), E-F(9) | **B-C(3)** | 10 |
| 5 | \{A, D, B, E, C\} | C-F(6), E-F(9) | **C-F(6)** | 16 |
| 6 | 全部 6 個點都在樹裡，停止 | — | — | **16** |

\*「跳」表示該邊的兩端已都在樹內，加進去會成環。

```python
import heapq

# graph = {'A': [('B', 4), ('D', 1)], ...}
def prim(graph, start):
    in_tree = set()
    heap = [(0, start)]  # (邊的權重, 節點)
    total = 0

    while heap:
        weight, node = heapq.heappop(heap)
        if node in in_tree:
            continue          # 已在樹裡，跳過
        in_tree.add(node)
        total += weight

        for neighbor, w in graph[node]:
            if neighbor not in in_tree:
                heapq.heappush(heap, (w, neighbor))  # 把新的候選邊加進去

    return total
```

時間複雜度：$O(|E| \log |V|)$，使用優先佇列（min-heap）。

### Kruskal 演算法

**策略：把所有邊按權重由小到大排序，依序考慮每條邊——加進去不會成環就加，否則跳過。**

與 Prim 不同，Kruskal 不從特定節點出發，而是全局地從最小邊開始挑，讓各個連通分量逐步合併。

判斷成環與否使用 **Union-Find（並查集）**：快速回答「兩節點是否已連通」——若已連通則加入會成環，否則合併。

<KruskalVisualizer client:only="svelte" />

#### 手把手：按邊權重由小到大考慮

邊排序後：A-D(1), B-E(2), B-C(3), A-B(4), C-E(5), C-F(6), D-E(7), A-E(8), E-F(9)

| 步驟 | 邊 | 兩端是否已連通 | 動作 | 分量狀態 |
| :---: | :--- | :---: | :---: | :--- |
| 1 | A-D(1) | 否 | **加** | \{A,D\}, \{B\}, \{C\}, \{E\}, \{F\} |
| 2 | B-E(2) | 否 | **加** | \{A,D\}, \{B,E\}, \{C\}, \{F\} |
| 3 | B-C(3) | 否 | **加** | \{A,D\}, \{B,E,C\}, \{F\} |
| 4 | A-B(4) | 否 | **加** | \{A,D,B,E,C\}, \{F\} |
| 5 | C-E(5) | 是 | 跳 | \{A,D,B,E,C\}, \{F\} |
| 6 | C-F(6) | 否 | **加** | \{A,D,B,E,C,F\} |
| — | 剩 1 個分量，停止 | — | — | — |

總權重：$1 + 2 + 3 + 4 + 6 = 16$

```python
# Union-Find：追蹤每個節點屬於哪個集合
parent = {}

def find(x):
    while parent[x] != x:
        parent[x] = parent[parent[x]]  # 路徑壓縮，讓樹更扁
        x = parent[x]
    return x

def union(x, y):
    rx, ry = find(x), find(y)
    if rx == ry:
        return False        # 已連通 → 加入會成環
    parent[ry] = rx         # 合併兩個集合
    return True

# edges = [('A', 'D', 1), ('B', 'E', 2), ...]
def kruskal(nodes, edges):
    for n in nodes:
        parent[n] = n                   # 每個節點初始自成一集合

    edges.sort(key=lambda e: e[2])      # 按權重由小到大排序
    total = 0

    for u, v, w in edges:
        if union(u, v):                 # 不成環才納入 MST
            total += w

    return total
```

時間複雜度：$O(|E| \log |E|)$，主要來自排序。

### Sollin 演算法

**策略：每一輪，所有連通分量各自找自己最便宜的跨分量邊，一次全部加進去，重複直到只剩一個分量。**

Sollin（又稱 Borůvka）是三個 MST 演算法中最早被發明的（1926 年），也是唯一天生適合平行運算的——每個分量可同時找自己的最小邊。

比較：Prim 從一棵樹逐步長大，Kruskal 逐條挑邊，Sollin 是所有分量同時行動、每輪合併。

<SollinVisualizer client:only="svelte" />

#### 手把手：每輪各分量同時選最小邊

**初始**：每個節點自成一個分量 → \{A\}, \{B\}, \{C\}, \{D\}, \{E\}, \{F\}

**第一輪**：每個分量挑自己最小的跨分量邊

| 分量 | 最小外連邊 |
| :---: | :---: |
| \{A\} | A-D(1) |
| \{B\} | B-E(2) |
| \{C\} | B-C(3) |
| \{D\} | A-D(1) |
| \{E\} | B-E(2) |
| \{F\} | C-F(6) |

本輪納入 MST 的邊（去重）：$\{\text{A-D}(1),\ \text{B-E}(2),\ \text{B-C}(3),\ \text{C-F}(6)\}$，總權重目前 12。

合併後分量：\{A, D\} 與 \{B, C, E, F\}

**第二輪**：兩個分量各自找最小跨分量邊

| 分量 | 外連邊候選 | 最小 |
| :---: | :--- | :---: |
| \{A, D\} | A-B(4), A-E(8), D-E(7) | **A-B(4)** |
| \{B, C, E, F\} | A-B(4), A-E(8), D-E(7) | **A-B(4)** |

納入 A-B(4)，合併為一個分量 → 停止。

總權重：$1 + 2 + 3 + 6 + 4 = 16$

```python
def sollin(nodes, edges):
    parent = {n: n for n in nodes}

    def find(x):
        while parent[x] != x:
            parent[x] = parent[parent[x]]
            x = parent[x]
        return x

    def union(x, y):
        rx, ry = find(x), find(y)
        if rx == ry:
            return False
        parent[ry] = rx
        return True

    total = 0
    num_components = len(nodes)

    while num_components > 1:
        # 每個分量找最小的跨分量邊
        cheapest = {}
        for u, v, w in edges:
            ru, rv = find(u), find(v)
            if ru == rv:
                continue  # 同分量，跳過
            for r in [ru, rv]:
                if r not in cheapest or w < cheapest[r][2]:
                    cheapest[r] = (u, v, w)

        # 把這些邊加進 MST
        for u, v, w in cheapest.values():
            if union(u, v):
                total += w
                num_components -= 1

    return total
```

時間複雜度：$O(|E| \log |V|)$。每輪至少讓分量數量減半（因為每個分量都會跟至少一個鄰居合併），所以最多 $\lceil \log |V| \rceil$ 輪。

### 三種 MST 演算法比較

| | Prim | Kruskal | Sollin |
| :--- | :--- | :--- | :--- |
| 方向 | 從節點出發，逐步擴張 | 從邊出發，合併集合 | 每輪每個分量各挑最小邊 |
| 適合 | 稠密圖（邊多） | 稀疏圖（邊少） | 平行運算場景 |
| 需要 | 優先佇列 | Union-Find + 排序 | Union-Find |
| 複雜度 | $O(|E| \log |V|)$ | $O(|E| \log |E|)$ | $O(|E| \log |V|)$ |
| 特色 | 直觀好理解 | 實作簡潔 | 天生可平行化 |

三個演算法的 MST 總權重一定相同（若有多棵 MST，選到的具體邊可能不同）。

## 單一起點之最短路徑

另一類經典的貪婪問題是**單源最短路徑（Single-Source Shortest Path）**：從一個起點出發，找到到達圖中所有其他節點的最短路徑。

### Dijkstra 演算法

**貪婪策略：每次從「還沒確定最短距離的節點」裡，選出目前距離起點最近的那個，把它標為「已確定」，再透過它更新鄰居的距離。**

「每次選最近的」即為貪婪策略。正確性前提：**無負權重邊**（有負邊需改用 Bellman-Ford）。

<DijkstraDiagram client:visible />

以上圖為例，從 A 出發：

| 步驟 | 已確定 | dist\[B\] | dist\[C\] | dist\[D\] | dist\[E\] |
| :---: | :--- | :---: | :---: | :---: | :---: |
| 初始 | \{A\} | 2 | 4 | ∞ | ∞ |
| 選 B(2) | \{A,B\} | 2 | 4 | 5 | ∞ |
| 選 C(4) | \{A,B,C\} | 2 | 4 | 5 | 9 |
| 選 D(5) | \{A,B,C,D\} | 2 | 4 | 5 | 9 |
| 選 E(9) | \{A,B,C,D,E\} | 2 | 4 | 5 | 9 |

```python
import heapq

# graph = {'A': [('B', 2), ('C', 4)], ...}
def dijkstra(graph, start):
    dist = {node: float('inf') for node in graph}
    dist[start] = 0          # 起點到自己距離為 0
    heap = [(0, start)]      # (目前距離, 節點)

    while heap:
        d, node = heapq.heappop(heap)
        if d > dist[node]:
            continue         # 這筆紀錄已過時，跳過

        for neighbor, weight in graph[node]:
            new_dist = d + weight
            if new_dist < dist[neighbor]:   # 找到更短的路，更新
                dist[neighbor] = new_dist
                heapq.heappush(heap, (new_dist, neighbor))

    return dist  # {'A': 0, 'B': 2, 'C': 4, 'D': 5, 'E': 9}
```

時間複雜度：$O((|V| + |E|) \log |V|)$，使用優先佇列。

:::warning
Dijkstra 不適用於含有負權重邊的圖。有負邊要改用 Bellman-Ford（$O(|V| \cdot |E|)$），它不是貪婪，而是反覆對所有邊做鬆弛。
:::

### 有向圖 vs 無向圖建法

Dijkstra 有向和無向都能跑，差別只在建圖方式：

```python
# 有向圖：A → B，只能單向走
graph['A'].append(('B', weight))

# 無向圖：A - B，雙向都能走
graph['A'].append(('B', weight))
graph['B'].append(('A', weight))
```

### 樹上路徑

樹的任意兩點間路徑唯一，不需要 Dijkstra——直接 BFS 從起點找到終點就行了：

```python
from collections import deque

# tree = {'A': [('B', 3), ('C', 5)], ...}
def tree_distance(tree, start, end):
    queue = deque([(start, 0)])  # (目前節點, 累計距離)
    visited = {start}

    while queue:
        node, dist = queue.popleft()
        if node == end:
            return dist
        for neighbor, weight in tree[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append((neighbor, dist + weight))
```

## 凸包（Convex Hull）

**問題**：給定平面上一組點，找出最小凸多邊形使所有點在其邊界或內部。直觀類比：用橡皮筋圈住所有釘子，收縮後的形狀即為凸包。

凸包是計算幾何的基礎，為碰撞偵測、路徑規劃、輪廓提取等進階問題的前置步驟。

### Graham Scan

Graham Scan 是最常見的凸包演算法：

1. 找到 $y$ 座標最小的點作為起點（$y$ 相同則取 $x$ 最小的）
2. 其他所有點按照相對於起點的極角排序（極角相同則按距離排序）
3. 依序掃描，維護一個堆疊——如果新點造成「右轉」，就把堆疊頂端的點彈出（它不是凸包點），直到形成「左轉」再推入新點

「左轉右轉」怎麼判斷？用**外積（cross product）**：給定三個點 $P_1, P_2, P_3$，

$$\text{cross} = (P_2.x - P_1.x)(P_3.y - P_1.y) - (P_2.y - P_1.y)(P_3.x - P_1.x)$$

- $\text{cross} > 0$：左轉（逆時針），保留
- $\text{cross} \leq 0$：右轉或共線，彈出 $P_2$

```python
import math

def graham_scan(points):
    # 找最下方的點作為起點
    start = min(points, key=lambda p: (p[1], p[0]))
    points.remove(start)

    # 按極角排序
    def polar_angle(p):
        return math.atan2(p[1] - start[1], p[0] - start[0])

    points.sort(key=polar_angle)

    stack = [start]
    for p in points:
        while len(stack) >= 2:
            cross = ((stack[-1][0] - stack[-2][0]) * (p[1] - stack[-2][1])
                   - (stack[-1][1] - stack[-2][1]) * (p[0] - stack[-2][0]))
            if cross <= 0:
                stack.pop()  # 右轉，不是凸包點
            else:
                break
        stack.append(p)

    return stack
```

時間複雜度：$O(n \log n)$，主要來自排序。每個點最多被推入和彈出堆疊一次，所以掃描部分是 $O(n)$。

#### 手把手：5 個點走一遍

5 個點：$P_0=(0,0),\ P_1=(4,0),\ P_2=(5,3),\ P_3=(2,4),\ P_4=(2,2)$（$P_4$ 位於內部）

- 起點：$P_0$（$y$ 最小）
- 按極角排序後順序：$P_1 \to P_2 \to P_4 \to P_3$

| 步驟 | 處理點 | 堆疊檢查（三點轉向） | 外積 | 動作 | 堆疊 |
| :---: | :---: | :--- | :---: | :--- | :--- |
| 0 | — | — | — | 初始 | \[P0\] |
| 1 | P1 | — | — | 直接推 | \[P0, P1\] |
| 2 | P2 | P0 → P1 → P2 | $+12$ | 左轉，推 | \[P0, P1, P2\] |
| 3 | P4 | P1 → P2 → P4 | $+8$ | 左轉，推 | \[P0, P1, P2, P4\] |
| 4 | P3 | P2 → P4 → P3 | $-6$ | 右轉，**彈 P4** | \[P0, P1, P2\] |
| 4 | P3 | P1 → P2 → P3 | $+10$ | 左轉，推 | \[P0, P1, P2, P3\] |

最終凸包：$P_0, P_1, P_2, P_3$（$P_4$ 被正確彈出，因為它在內部）。

### Jarvis March（Gift Wrapping）

另一種貪婪做法。從最左邊的點開始，每次找「最外側」的下一個點，一圈繞回來。

時間複雜度是 $O(nh)$，$h$ 是凸包上的點數。當 $h$ 很小的時候比 Graham Scan 快，但最壞情況是 $O(n^2)$。

## 霍夫曼編碼（Huffman Coding）

資料壓縮的經典應用。核心策略：**出現頻率高的字元用短編碼，頻率低的用長編碼**，使整體編碼長度最短。

### 為什麼不能直接用固定長度

A、B、C、D 四個字元的固定長度編碼各需 2 bit。但若 A 出現 1000 次、D 只出現 1 次，等長編碼很浪費。

可變長度編碼的問題在於斷詞歧義——「01011」是 0-1-0-1-1 還是 01-01-1？Huffman 使用**前綴碼（Prefix Code）** 解決：沒有任何字元的編碼是另一個字元的前綴，解碼時不會有歧義。

### 建構 Huffman Tree

建構過程本身就是貪婪的：

1. 每個字元建一個節點，權重就是頻率
2. 反覆取出權重最小的兩個節點，合併成一個新節點，新權重 = 兩個子節點權重之和
3. 重複直到只剩一個節點——就是 Huffman Tree 的根

走完後，從根到每個葉子的路徑就是該字元的編碼：往左走是 0，往右走是 1。

<HuffmanTreeDiagram client:only="svelte" />

以上圖的例子，最終編碼結果：

| 字元 | 頻率 | 編碼 | 位元數 |
| :---: | :---: | :--- | :---: |
| F | 45 | 0 | 1 |
| C | 12 | 100 | 3 |
| D | 13 | 101 | 3 |
| A | 5 | 1100 | 4 |
| B | 9 | 1101 | 4 |
| E | 16 | 111 | 3 |

F 最常出現所以編碼最短（1 bit），A 最少出現所以編碼最長（4 bit）。

加權路徑長度 = $45 \times 1 + 12 \times 3 + 13 \times 3 + 5 \times 4 + 9 \times 4 + 16 \times 3 = 224$ bit。如果用固定 3 bit 編碼全部字元，需要 $100 \times 3 = 300$ bit。Huffman 省了 25% 以上。

```python
import heapq

def huffman(freq):
    # freq = {'A': 5, 'B': 9, 'C': 12, 'D': 13, 'E': 16, 'F': 45}
    heap = [(f, i, c) for i, (c, f) in enumerate(freq.items())]
    heapq.heapify(heap)
    counter = len(freq)
    tree = {}

    while len(heap) > 1:
        f1, _, left = heapq.heappop(heap)
        f2, _, right = heapq.heappop(heap)
        merged = (left, right)
        heapq.heappush(heap, (f1 + f2, counter, merged))
        counter += 1

    # 從樹根生成編碼
    def build_codes(node, prefix=''):
        if isinstance(node, str):
            tree[node] = prefix
            return
        build_codes(node[0], prefix + '0')
        build_codes(node[1], prefix + '1')

    build_codes(heap[0][2])
    return tree
```

時間複雜度：$O(n \log n)$，$n$ 是字元數。每次從 heap 取出和放入都是 $O(\log n)$，共 $n - 1$ 輪。

:::info
Huffman 編碼是 **最優前綴碼**——在所有前綴碼中，它的加權路徑長度最短。這可以用交換論證證明：如果有一個更好的編碼，你可以把它的兩個最深葉子換成 Huffman 的選擇，不會變更差。
:::

## 活動選擇問題

一間會議室，多場會議各有開始與結束時間，目標是在不衝突的前提下選出最多場次。

| 會議 | 開始 | 結束 |
| :--- | :---: | :---: |
| A | 1 | 4 |
| B | 3 | 5 |
| C | 0 | 6 |
| D | 5 | 7 |
| E | 3 | 9 |
| F | 6 | 10 |
| G | 8 | 11 |

貪婪策略：**按結束時間由早到晚排序，每次選最早結束且不衝突的那場。**

理由：結束越早，留給後續的空間越多。

選擇過程：A → B 衝突跳過 → D → C、F 衝突跳過 → G。結果：A、D、G，共 3 場。

時間複雜度：$O(n \log n)$，排序主導。

## 貪婪下的打包問題（Bin Packing）

不同大小的物品需塞入固定容量的箱子，目標是用最少箱子數。此問題為 NP-hard，但貪婪策略可在多項式時間內給出近似解。

<BinPackingDiagram client:only="svelte" />

### 常見的貪婪策略

**Next Fit（下一個適合就放）**

只看當前箱子，放得下就放，放不下就開新箱。最簡單但最浪費——不會回頭利用前面箱子的剩餘空間。

**First Fit（第一個適合就放）**

從第一個箱子開始找，找到第一個放得下的就放進去。比 Next Fit 好，因為會利用前面的空間。

**First Fit Decreasing（先排序再塞）**

先把物品從大到小排序，再用 First Fit。大東西先塞進去，小東西用來填縫隙，效果通常最好。

```python
def first_fit_decreasing(items, capacity=1.0):
    items_sorted = sorted(items, reverse=True)
    bins = []

    for item in items_sorted:
        placed = False
        for b in range(len(bins)):
            if bins[b] + item <= capacity:
                bins[b] += item
                placed = True
                break
        if not placed:
            bins.append(item)

    return len(bins)
```

### 近似比

| 策略 | 近似比 | 直覺 |
| :--- | :---: | :--- |
| Next Fit | $\leq 2 \cdot OPT$ | 最粗暴 |
| First Fit | $\leq 1.7 \cdot OPT$ | 回頭找空間 |
| First Fit Decreasing | $\leq \frac{11}{9} \cdot OPT + \frac{6}{9}$ | 先排序再塞，效果最好 |

這裡 $OPT$ 是最優解的箱子數。FFD 的近似比約 1.22，意思是它最多比最優解多用 22% 的箱子——在大多數實際應用中已經夠好了。

:::tip
Bin Packing 跟找零問題很類似——貪婪不保證最優，但能快速給出接近最優的答案。需要精確最優解時得用動態規劃或整數線性規劃。
:::

## 貪婪的常見陷阱：0/1 背包

背包容量有限，每件物品只能整件拿或不拿，目標為總價值最大。

| 物品 | 重量 | 價值 | 性價比 |
| :--- | :---: | :---: | :---: |
| A | 1 | 6 | 6.0 |
| B | 3 | 10 | 3.3 |
| C | 4 | 12 | 3.0 |

背包容量 5。

- 貪婪（按性價比）：選 A + B，總價值 **16**
- 最優解：選 A + C，總價值 **18**

貪婪給了錯誤答案。0/1 背包需要動態規劃。

:::tip
若物品可以切割（分數背包），貪婪可得到最優解。0/1 背包（整件拿或不拿）則不行，因為貪婪選擇性質不成立。
:::

## 貪婪 vs 動態規劃

| | 貪婪 | 動態規劃 |
| :--- | :--- | :--- |
| 決策方式 | 每步選局部最優，不回頭 | 考慮所有可能，記錄子問題答案 |
| 額外條件 | 需滿足貪婪選擇性質 | 只需最優子結構 |
| 速度 | 通常更快 | 通常較慢 |
| 正確性 | 需要證明 | 狀態設計正確就保證正確 |
| 典型例子 | 活動選擇、MST、Dijkstra、Huffman | 0/1 背包、最長公共子序列 |

貪婪通常簡單且快速，難點在於判斷其在特定問題上是否成立。
