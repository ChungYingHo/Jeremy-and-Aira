上一篇淘汰與搜尋法的核心動作是「縮小」——每一步決定哪一塊不用看、丟掉它。這篇講的分而治之（Divide and Conquer）看起來也在縮小，但意思完全不一樣：切完後**每一塊都要解**，最後再把答案合回去。兩者很容易混淆，所以這篇一開頭先把差異講清楚，再一路走完排序、幾何、FFT 五個經典應用。

## 分而治之概論

分而治之的標準三步驟：

1. **Divide**：把大問題切成若干規模較小、結構相同的子問題
2. **Conquer**：對每個子問題遞迴求解（子問題小到不能再切時，直接解）
3. **Combine**：把子問題的解合併成原問題的解

關鍵詞是**合併**。沒有合併就不是分治——那是淘汰。

### 與淘汰與搜尋法的差異

兩者都「切成小塊」，但後續動作完全不同：

| 面向 | 淘汰與搜尋 | 分而治之 |
| :--- | :--- | :--- |
| 切完後 | 只進**其中一塊**，其他丟棄 | **每一塊**都要解 |
| 合併 | 不需要（答案只在一塊裡） | 必要，常是核心步驟 |
| 典型遞迴式 | $T(n) = T(n/c) + f(n)$ | $T(n) = a\,T(n/b) + f(n)$ |
| 常見複雜度 | $O(\log n)$ 或 $O(n)$ | $O(n \log n)$ 居多 |
| 代表 | 二元搜尋、偽幣、第 $k$ 小 | 合併排序、FFT |

比喻：淘汰是「問路——哪邊沒有寶藏？那邊跳過」；分治是「分組尋寶——每組各自找，找到後回報合起來看」。

### 基本遞迴式與直覺

分治的遞迴式長這樣：

$$T(n) = a\,T(n/b) + f(n)$$

- $a$：切完後有幾個子問題要解
- $b$：每個子問題的規模相對原問題縮小幾倍
- $f(n)$：把 $a$ 個子問題的解合併起來的成本

典型例子：

- **合併排序**：切成 2 份，各自解，合併花 $O(n)$ → $T(n) = 2T(n/2) + O(n) = O(n \log n)$
- **二元搜尋（淘汰）**：切成 2 份只解 1 份，合併花 $O(1)$ → $T(n) = T(n/2) + O(1) = O(\log n)$

能不能達到 $O(n \log n)$，取決於「合併成本」是否為線性。合併若比線性大，整體就會被合併拖垮。

### 什麼問題適合分治

- 問題有「自相似結構」：子問題跟原問題長一樣，只是規模小
- 子問題之間**獨立**（否則要動態規劃，下一系列再談）
- 合併成本合理（通常要 $O(n)$ 或更低）

接下來五個範例，從最熟悉的排序開始，逐步走到幾何與訊號處理。

## 合併排序法

排序篇已經看過合併排序的實作，這裡從分治三步驟重新看它的骨架。

### 三步驟對應

1. **Divide**：把陣列從中間切成左右兩半
2. **Conquer**：對左右兩半各自遞迴呼叫合併排序
3. **Combine**：把兩個**已排序**的子陣列合併成一個排序好的大陣列

重點在第 3 步的「已排序」——因為遞迴保證子陣列是有序的，合併時才能用「兩個指標各取最小」的線性方法。

### 合併步驟細拆

給兩個已排序陣列 $L$、$R$，合併成一個排序好的 $M$：

1. 設兩個指標 $i = 0$、$j = 0$，結果陣列 $M$ 為空
2. 比較 $L[i]$ 與 $R[j]$：
   - 誰小就把誰放進 $M$，該指標 +1
3. 重複步驟 2 直到其中一邊取完
4. 把另一邊剩下的元素直接接到 $M$ 尾端

因為兩邊都已排序，**每個元素只被看過一次**，合併成本 $O(n)$。

### 手把手走一遍：[4, 1, 3, 1, 5, 2]

**Divide 階段**：不斷從中間切半，直到每塊剩一個元素。`[4, 1, 3, 1, 5, 2]` → `[4, 1, 3]` + `[1, 5, 2]` → 各自再切 → 最終剩 `[4]`、`[1]`、`[3]`、`[1]`、`[5]`、`[2]` 六個單元素。單一元素本身就是排序好的，從這裡開始往上合併。

**Combine 階段**（兩兩合併）：

| 步驟 | 合併對象 | 結果 |
| :---: | :--- | :--- |
| 1 | `[1]` + `[3]` | `[1, 3]` |
| 2 | `[4]` + `[1, 3]` | `[1, 3, 4]` |
| 3 | `[5]` + `[2]` | `[2, 5]` |
| 4 | `[1]` + `[2, 5]` | `[1, 2, 5]` |
| 5 | `[1, 3, 4]` + `[1, 2, 5]` | `[1, 1, 2, 3, 4, 5]` |

最後一步的合併細節：

| $i$ | $j$ | $L[i]$ | $R[j]$ | 動作 | $M$ |
| :---: | :---: | :---: | :---: | :--- | :--- |
| 0 | 0 | 1 | 1 | $L$ 不大於 $R$，取 $L$ | `[1]` |
| 1 | 0 | 3 | 1 | $R$ 較小，取 $R$ | `[1, 1]` |
| 1 | 1 | 3 | 2 | $R$ 較小，取 $R$ | `[1, 1, 2]` |
| 1 | 2 | 3 | 5 | $L$ 較小，取 $L$ | `[1, 1, 2, 3]` |
| 2 | 2 | 4 | 5 | $L$ 較小，取 $L$ | `[1, 1, 2, 3, 4]` |
| 3 | 2 | — | 5 | $L$ 用完，把 $R$ 剩下全接上 | `[1, 1, 2, 3, 4, 5]` |

「$L$ 不大於 $R$ 時取 $L$」的細節很重要——這讓相同鍵值的元素保留原本順序，也就是為什麼合併排序是**穩定排序**。

### 複雜度

每層切半 → 共 $\log n$ 層；每層合併花 $O(n)$：

$$T(n) = 2T(n/2) + O(n) = O(n \log n)$$

任何輸入都保證 $O(n \log n)$，代價是需要 $O(n)$ 額外空間來放合併結果。

## 快速排序法

快速排序也是分治，但「切法」與「合併」跟合併排序完全相反。

### 三步驟對應

1. **Divide**：挑一個 pivot，把陣列分成「小於 pivot」「等於 pivot」「大於 pivot」三堆
2. **Conquer**：對「小於」和「大於」兩堆各自遞迴
3. **Combine**：把左側結果、pivot、右側結果**直接接在一起**

注意 Combine 的特殊性——幾乎**不做事**。因為 Divide 已經按大小分好了，遞迴回來的結果拼起來就已經是排序好的。

對比：

| | Divide | Combine |
| :--- | :--- | :--- |
| 合併排序 | 便宜（取中點） | 昂貴（要真的比對合併） |
| 快速排序 | 昂貴（掃一遍分堆） | 便宜（直接接上） |

兩者把「分」與「合」的工作量剛好交換，但總合都是 $O(n \log n)$（平均情況）。

### 手把手走一遍：[4, 1, 3, 1, 5, 2]

每一層挑中間元素當 pivot。

**第一層**：陣列 `[4, 1, 3, 1, 5, 2]`，pivot = `arr[3] = 1`

- 小於 1：`[]`
- 等於 1：`[1, 1]`
- 大於 1：`[4, 3, 5, 2]`

對左右兩堆遞迴。左邊空直接完成，右邊繼續切。

**第二層（右）**：`[4, 3, 5, 2]`，pivot = `arr[2] = 5`

- 小於 5：`[4, 3, 2]`
- 等於 5：`[5]`
- 大於 5：`[]`

**第三層**：`[4, 3, 2]`，pivot = `arr[1] = 3`

- 小於 3：`[2]`
- 等於 3：`[3]`
- 大於 3：`[4]`

三堆各自只剩一個元素，直接回傳。

**由內往外 Combine**（全部直接串接）：

| 層 | 左 | 中（pivot） | 右 | 合併結果 |
| :---: | :--- | :---: | :--- | :--- |
| 3 | `[2]` | `[3]` | `[4]` | `[2, 3, 4]` |
| 2 | `[2, 3, 4]` | `[5]` | `[]` | `[2, 3, 4, 5]` |
| 1 | `[]` | `[1, 1]` | `[2, 3, 4, 5]` | `[1, 1, 2, 3, 4, 5]` |

### 最壞情況退化

如果 pivot 每次都挑到最大或最小值，一邊會是空的、另一邊是 $n-1$ 個元素：

$$T(n) = T(n-1) + O(n) = O(n^2)$$

已排序輸入 + 固定挑第一個當 pivot 最容易觸發。實務做法：

- **隨機 pivot**：從陣列中隨機抽一個，期望情況仍為 $O(n \log n)$
- **三數取中（median-of-three）**：取首、中、尾三個的中位數當 pivot，對「幾乎有序」的輸入特別有效

這兩個技巧把「最壞情況的機率」壓到可忽略的程度，實務上多數語言標準庫的排序底層就是混用 Quick Sort + Insertion Sort（小陣列切換，如 Introsort、Timsort）。

## 二維平面上的極點

問題定義：

- 給定平面上 $n$ 個點
- 找出這些點的**凸包（Convex Hull）**——包住所有點的最小凸多邊形上的頂點

直覺想像：把所有點釘在板上，拿一條橡皮筋從外面套起來鬆開，橡皮筋形成的多邊形頂點就是極點。

### Quickhull：分治版本的凸包演算法

名字致敬 Quick Sort——兩者同樣靠「用一條線分兩邊、遞迴處理」。

### 演算法步驟

1. 在所有點中找**最左**的點 $A$ 與**最右**的點 $B$（若 x 相同，取 y 較小 / 較大）
2. $A$ 與 $B$ 必為凸包頂點（最外側不可能被夾住）
3. 線段 $AB$ 把其餘點分成兩組：
   - **上半**：在 $AB$ 之上的點
   - **下半**：在 $AB$ 之下的點
4. 對上半與下半各自呼叫 `findHull`

`findHull(points, P, Q)`：找出線段 $PQ$ 上側的凸包點

1. 若 `points` 為空，結束（$PQ$ 直接是凸包的一條邊）
2. 從 `points` 中找**離 $PQ$ 最遠**的點 $C$
3. $C$ 必為凸包頂點（它比線段 $PQ$ 更外側）
4. 三角形 $PCQ$ 內部的點都**不是**凸包頂點（被包住了），直接丟掉
5. 剩下的點分成兩組：
   - $PC$ 外側的點 → 遞迴呼叫 `findHull(group1, P, C)`
   - $CQ$ 外側的點 → 遞迴呼叫 `findHull(group2, C, Q)`

三角形 $PCQ$ 是關鍵——它一次淘汰掉所有「被包在裡面」的點，只留下可能還是頂點的外側點繼續往下找。

### 點到線段的位置怎麼判

「在 $AB$ 之上」「離 $PQ$ 最遠」這兩件事都靠**叉積（cross product）**：

給三點 $P = (x_1, y_1)$、$Q = (x_2, y_2)$、$R = (x_3, y_3)$，定義：

$$d = (x_2 - x_1)(y_3 - y_1) - (y_2 - y_1)(x_3 - x_1)$$

- $d > 0$：$R$ 在 $PQ$ **左側**
- $d < 0$：$R$ 在 $PQ$ **右側**
- $d = 0$：三點共線
- $|d|$ 的大小**正比**於 $R$ 到 $PQ$ 的距離

這是凸包、線段相交、點在多邊形內等幾何演算法的共用工具，記下來就能反覆用。

### 手把手走一遍

給 8 個點：`P1=(0,3)`、`P2=(2,5)`、`P3=(4,7)`、`P4=(6,4)`、`P5=(8,2)`、`P6=(5,1)`、`P7=(3,2)`、`P8=(1,1)`。

**步驟 1**：最左 = $P1 = (0, 3)$，最右 = $P5 = (8, 2)$ → 必為凸包頂點。

**步驟 2**：以線段 $P1P5$ 分上下。

- 上半：$P2, P3, P4$
- 下半：$P6, P7, P8$

**步驟 3**（上半遞迴）`findHull([P2, P3, P4], P1, P5)`：

- 離 $P1P5$ 最遠：$P3 = (4, 7)$ → 加入凸包
- 三角形 $P1 P3 P5$ 內部：$P4$ 在三角形內（被包住）→ 丟棄
- 剩下 $P2$ 在 $P1P3$ 外側 → 遞迴 `findHull([P2], P1, P3)`

`findHull([P2], P1, P3)`：只剩 $P2$，離 $P1P3$ 稍遠（實際算叉積判斷），若在外側 → $P2$ 加入凸包。繼續往下遞迴會得到空集合 → 結束。

**步驟 4**（下半遞迴）`findHull([P6, P7, P8], P5, P1)`：

- 離 $P5P1$ 最遠：$P8 = (1, 1)$ → 加入凸包
- 三角形 $P5 P8 P1$ 內部：$P7$ 在三角形內 → 丟棄
- 剩下 $P6$ 在 $P5P8$ 外側 → 遞迴加入 $P6$

最終凸包頂點（依序）：$P1 \to P2 \to P3 \to P5 \to P6 \to P8 \to P1$

### 複雜度

- 平均情況：每次能切掉約一半的點 → $T(n) = 2T(n/2) + O(n) = O(n \log n)$
- 最壞情況：點幾乎都在凸包上、每次三角形幾乎切不掉東西 → $O(n^2)$

想法與 Quick Sort 完全平行：平均很好、最壞會退化，但實務輸入很少觸發最壞情況。

其他經典凸包演算法還有 Graham Scan、Andrew's Monotone Chain，穩定保證 $O(n \log n)$，排序為主體。Quickhull 是「分治」這個框架下最乾淨的範例。

## 最接近的兩點

問題定義：

- 給定平面上 $n$ 個點
- 找出距離最近的一對點，回傳它們的距離

Naive 做法：兩兩配對計算距離，$O(n^2)$。分治版本可以壓到 $O(n \log n)$——而且這題是「分治設計的經典教材」，因為它的合併步驟需要一些巧思。

### 演算法步驟

1. 預先把所有點按 x 座標排序一次
2. 呼叫 `closest(points)`：
   - 若 `points` 數量小（例如 $\leq 3$），直接兩兩比對找最短距離
   - 否則：
     - **Divide**：在中位 x 座標切開，切成左半 $P_L$ 與右半 $P_R$，各約 $n/2$ 個點
     - **Conquer**：$d_L = \text{closest}(P_L)$、$d_R = \text{closest}(P_R)$
     - 取 $d = \min(d_L, d_R)$
     - **Combine**：檢查「跨越中線」的點對是否比 $d$ 更近

Combine 是難點——跨越中線的可能點對最多有 $n_L \times n_R \approx n^2/4$ 組，若全配對就退回 $O(n^2)$ 了。

### Combine 的關鍵剪枝：strip

設中線在 $x = x_m$。**只有**左右距離中線都在 $d$ 以內的點，才有可能跟對面的點配出更短距離（超過 $d$ 的點跨過中線至少就有 $d$ 的距離了）。

1. 從原點集中篩出所有 $|x - x_m| < d$ 的點，這些點組成「strip」
2. 把 strip 內的點按 **y 座標**排序
3. 對 strip 內每個點 $p$：**只檢查接下來 y 座標在 $p.y + d$ 範圍內的點**（實務上最多 7 個）
4. 若找到任一對距離 $< d$，更新 $d$

為什麼每個點只需檢查後面 7 個？因為 strip 的寬是 $2d$，而左右兩邊各自內部最近距離都 $\geq d$，幾何上可以證明 strip 內**任一 $d \times 2d$ 的矩形**至多只能放 8 個點——再多就違反「同側內部距離 $\geq d$」這個前提。

記住結論就好：「y 排序後，每個點往下掃不超過 7 個」即可把 Combine 壓到 $O(n)$。

### 手把手走一遍

給 6 個點（已按 x 排序）：`A=(1,3)`、`B=(2,1)`、`C=(3,4)`、`D=(5,2)`、`E=(6,5)`、`F=(8,3)`。

**第一層 Divide**：中位 x 約為 4，切成左半 `[A, B, C]` 與右半 `[D, E, F]`。

**左半 `[A, B, C]`**：3 點直接兩兩算

| 對 | 距離 |
| :---: | :---: |
| $AB$ | $\sqrt{(1-2)^2 + (3-1)^2} = \sqrt{5} \approx 2.24$ |
| $AC$ | $\sqrt{(1-3)^2 + (3-4)^2} = \sqrt{5} \approx 2.24$ |
| $BC$ | $\sqrt{(2-3)^2 + (1-4)^2} = \sqrt{10} \approx 3.16$ |

左半最近距離 $d_L \approx 2.24$。

**右半 `[D, E, F]`**：3 點直接兩兩算

| 對 | 距離 |
| :---: | :---: |
| $DE$ | $\sqrt{1^2 + 3^2} = \sqrt{10} \approx 3.16$ |
| $DF$ | $\sqrt{3^2 + 1^2} = \sqrt{10} \approx 3.16$ |
| $EF$ | $\sqrt{2^2 + 2^2} = \sqrt{8} \approx 2.83$ |

右半最近距離 $d_R \approx 2.83$。

**Combine**：$d = \min(d_L, d_R) = 2.24$。中線 $x_m = 4$。

篩 strip（$|x - 4| < 2.24$ → x 落在 $(1.76, 6.24)$）：

- 左：$B = (2, 1)$、$C = (3, 4)$
- 右：$D = (5, 2)$、$E = (6, 5)$

按 y 排序：$B(y=1) \to D(y=2) \to C(y=4) \to E(y=5)$

檢查每個點與後續 y 差在 $d = 2.24$ 內的點：

| 起點 | 要檢查 | 距離 | 更新 $d$？ |
| :---: | :--- | :--- | :---: |
| $B(2,1)$ | $D(5,2)$（$\Delta y = 1$） | $\sqrt{9+1} = \sqrt{10} \approx 3.16$ | 否 |
| $B(2,1)$ | $C(3,4)$（$\Delta y = 3 > d$，跳過） | — | — |
| $D(5,2)$ | $C(3,4)$（$\Delta y = 2$） | $\sqrt{4+4} = \sqrt{8} \approx 2.83$ | 否 |
| $D(5,2)$ | $E(6,5)$（$\Delta y = 3 > d$，跳過） | — | — |
| $C(3,4)$ | $E(6,5)$（$\Delta y = 1$） | $\sqrt{9+1} \approx 3.16$ | 否 |

沒找到更近的 → 最終最近距離 $\approx 2.24$（$AB$ 或 $AC$）。

### 複雜度

$$T(n) = 2T(n/2) + O(n) = O(n \log n)$$

每層切半共 $\log n$ 層，每層 Combine 花 $O(n)$（strip 篩選 + y 排序 + 每點最多檢查 7 個）。

實作細節：若每次 Combine 都重新對 strip 做 y 排序，會退化成 $O(n \log^2 n)$。快速版本是在遞迴過程中**同時維護兩份預排序索引**（x 與 y），Combine 時直接過濾出 y 排序後的 strip。這個優化讓整體複雜度保持 $O(n \log n)$。

## 快速傅立葉轉換

問題背景（不用懂數學也沒關係）：

- 給兩個多項式 $A(x)$（$n$ 項）、$B(x)$（$n$ 項）
- 想算乘積 $C(x) = A(x) \cdot B(x)$
- 直接乘：每項都要配每項 → $O(n^2)$
- **快速傅立葉轉換（FFT）**能把它壓到 $O(n \log n)$

FFT 的數學原文很嚇人（單位根、複數、離散傅立葉轉換），但**它本身是分治**。這一節只講分治框架，數學證明略過。

### 為什麼多項式乘法可以變快

核心觀察是「多項式有兩種表示法」：

1. **係數表示法**：$A(x) = a_0 + a_1 x + a_2 x^2 + \ldots$，用 $[a_0, a_1, a_2, \ldots]$ 描述
2. **點值表示法**：只要給 $n+1$ 個點 $(x_i, A(x_i))$，$n$ 次多項式就被唯一決定

兩種表示法下的乘法成本：

| 表示法 | 乘法成本 |
| :---: | :---: |
| 係數表示法 | $O(n^2)$（逐項配對） |
| 點值表示法 | $O(n)$（對應點相乘即可） |

只要能快速**在兩種表示法間切換**，整個乘法就快了：

1. 把 $A$、$B$ 從係數表示轉成點值表示
2. 點值對應相乘（$O(n)$）
3. 把相乘結果從點值表示轉回係數表示

FFT 就是那個「係數 ↔ 點值」的快速轉換器。

### 分治怎麼切

目標：給多項式 $A(x) = a_0 + a_1 x + a_2 x^2 + \ldots + a_{n-1} x^{n-1}$，快速算出它在 $n$ 個特定點上的值。

**關鍵觀察**：按下標奇偶把係數分兩組：

$$A(x) = \underbrace{(a_0 + a_2 x^2 + a_4 x^4 + \ldots)}_{A_{\text{even}}(x^2)} + x \cdot \underbrace{(a_1 + a_3 x^2 + a_5 x^4 + \ldots)}_{A_{\text{odd}}(x^2)}$$

也就是說：

$$A(x) = A_{\text{even}}(x^2) + x \cdot A_{\text{odd}}(x^2)$$

$A_{\text{even}}$ 與 $A_{\text{odd}}$ 各自是規模 $n/2$ 的多項式。要算 $A(x_1), A(x_2), \ldots, A(x_n)$，只要先算出 $A_{\text{even}}$ 和 $A_{\text{odd}}$ 在 $x_1^2, x_2^2, \ldots$ 這些「平方後」的點上的值，再組合起來就好。

### 為什麼選單位根當 $x$

平方後的點數**還是 $n$ 個**，看起來沒變少。這就是「單位根」登場的地方——FFT 特別選 **$n$ 個複數單位根** $\omega_n^0, \omega_n^1, \ldots, \omega_n^{n-1}$ 當計算點。

單位根有一個神奇性質：**平方後的 $n$ 個值只有 $n/2$ 個不同值**（因為 $\omega^k$ 與 $\omega^{k+n/2}$ 平方後相等）。

這就是 FFT 的全部魔法：

- 原問題：算 $A$ 在 $n$ 個單位根的值（規模 $n$）
- 遞迴：算 $A_{\text{even}}$ 和 $A_{\text{odd}}$ 在 $n/2$ 個平方後單位根的值（規模各 $n/2$）
- 合併：用 $A(x) = A_{\text{even}}(x^2) + x \cdot A_{\text{odd}}(x^2)$ 組回來，$O(n)$

### 複雜度

$$T(n) = 2T(n/2) + O(n) = O(n \log n)$$

完美的分治遞迴式，跟合併排序一樣。

### 對初學者的總結

不用記單位根公式也沒關係，記住這些結論：

| 問題 | 不用 FFT | 用 FFT |
| :---: | :---: | :---: |
| 多項式乘法 | $O(n^2)$ | $O(n \log n)$ |
| 大整數乘法 | $O(n^2)$ | $O(n \log n)$ |
| 訊號卷積 / 濾波 | $O(n^2)$ | $O(n \log n)$ |

只要看到「兩個序列配對累加」的運算（卷積），FFT 幾乎都派得上用場。實務上 Python 的 `numpy.fft`、訊號處理的 `scipy.signal.fftconvolve`、壓縮軟體（MP3、JPEG 的近親 DCT）背後都是它。

FFT 的核心價值：

1. 分治把 $O(n^2)$ 砍到 $O(n \log n)$
2. 「選對計算點」讓 Combine 的成本保持線性
3. 抽象出來就是三步驟：$\text{係數} \to \text{點值} \to$ 點值相乘 $\to \text{點值} \to \text{係數}$

這是分治框架最戲劇性的例子——表面上是多項式，本質上是「把規模 $n$ 的計算拆成兩個規模 $n/2$」的純結構手法。

## 結論

分而治之是演算法設計的主幹之一，共同模式：

| 階段 | 做什麼 |
| :--- | :--- |
| Divide | 把問題切成規模較小、結構相同的子問題 |
| Conquer | 對子問題遞迴求解 |
| Combine | 把子問題的解合併成原問題的解 |

關鍵經驗：

- **Combine 決定整體效率**。Combine 做到 $O(n)$ 就能配上 $T(n) = 2T(n/2) + O(n) = O(n \log n)$
- **獨立子問題**是前提。子問題若會互相依賴，就要動態規劃
- 跟淘汰與搜尋法的差別就一句話：**分治每塊都要解並合起來，淘汰只走其中一條路**

五個範例的映射表：

| 問題 | Divide | Combine | 複雜度 |
| :--- | :--- | :--- | :---: |
| 合併排序 | 切半 | 兩指標合併 | $O(n \log n)$ |
| 快速排序 | 以 pivot 分三堆 | 直接接上 | 平均 $O(n \log n)$，最差 $O(n^2)$ |
| 凸包 Quickhull | 以線段分上下 | 遞迴結果接起來 | 平均 $O(n \log n)$ |
| 最接近兩點 | 以 x 中線切半 | strip + y 排序，每點查 7 個 | $O(n \log n)$ |
| FFT | 按係數奇偶分 | 用 $A(x) = A_e(x^2) + x A_o(x^2)$ 組合 | $O(n \log n)$ |

下一篇會進動態規劃——當子問題**不再獨立**、彼此重疊時，分治的遞迴會重複計算，動態規劃用「記下來不要重算」解掉這個低效，是另一條完全不同的思路。
