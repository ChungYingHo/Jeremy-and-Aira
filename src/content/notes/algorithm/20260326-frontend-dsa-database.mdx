## 資料與演算法比我們想像得更近

從一個你一定寫過的東西開始——從陣列裡把重複的值拿掉，也就是去重。

常見的寫法有三種：

```js
// 寫法一：土法煉鋼，用迴圈 + includes
function removeDuplicates(arr) {
  const result = []
  for (const item of arr) {
    // 每次都問：「result 裡面有沒有這個東西？」
    // includes 會從頭掃到尾，有 n 個元素就最多比 n 次
    if (!result.includes(item)) {
      result.push(item)
    }
  }
  return result
}
// 複雜度：O(n²)
// 外層迴圈跑 n 次，每次 includes 最多掃 n 次 → n × n
```

```js
// 寫法二：filter + indexOf
function removeDuplicates(arr) {
  // 只保留「第一次出現」的元素
  // indexOf 會回傳該值第一次出現的位置
  return arr.filter((item, index) => arr.indexOf(item) === index)
}
// 複雜度：O(n²)
// filter 跑 n 次，每次 indexOf 從頭掃到尾最多 n 次 → 一樣是 n × n
```

```js
// 寫法三：一行搞定
function removeDuplicates(arr) {
  return [...new Set(arr)]
}
// 複雜度：O(n)
// Set 內部用 hash table，加入和查找都是固定時間
// 整個陣列只需要跑一遍
```

三種寫法跑出來的結果完全一樣。但寫法一和寫法二都是 $O(n^2)$（資料量多 10 倍，慢 100 倍），寫法三是 $O(n)$（資料量多 10 倍，只慢 10 倍）。

### 用排隊來想就懂了

想像一下你在管理一個「不重複名單」。

**寫法一、二就像這樣：** 每來一個新名字，你就從名單的第一行開始，一行一行往下看：「有沒有這個名字？有沒有？有沒有？」名單上有 100 個名字，你就得看 100 行。下一個人來，又從頭看 100 行。再下一個人，又看 100 行⋯⋯

如果今天有 1000 個人排隊要登記，名單上也累積了快 1000 個名字——你每登記一個人就要掃一遍快 1000 行的名單。1000 個人 × 1000 行 = 一百萬次翻找。

**寫法三就像這樣：** 你手上有一本索引簿，每個名字都有專屬的格子。新名字來了，你直接翻到那一頁看看格子是空的還是有人——不用從頭掃，直接定位。不管名單上有多少人，查一個名字的速度都一樣快。

1000 個人 × 每次只查一格 = 只要 1000 次。

一百萬 vs 一千，差了一千倍。

其實 LeetCode 上有一題 [217. Contains Duplicate](https://leetcode.com/problems/contains-duplicate/)，就是判斷陣列裡有沒有重複值。如果你用 `includes` 或 `indexOf` 的邏輯去做，測資量一大就會直接 TLE（Time Limit Exceeded，超時）。換成 `Set` 就秒過——題目本身超簡單，考的就是你有沒有意識到不同做法的速度差異。

## 你的程式碼，到底有多慢？

上面那個「差了一千倍」的觀察，其實就是工程師說的「複雜度分析」在做的事。

不用怕，這不是數學課。我用生活的方式說給你聽。

### 想像你在找一本書

**情境一：書架上的書有編號**

你要找第 42 號的書。書架上每本書都有清楚的編號，你直接走到 42 號的位置拿走——不管書架上有 100 本還是 100 萬本，你都只需要走那一步。

這種「不管資料多少，速度都一樣」的情況，工程師會說它的速度等級是 **常數時間**。

**情境二：書架上的書沒有編號**

你要找一本叫《哈利波特》的書，但書架上的書是亂排的。你只好從第一本開始，一本一本翻：「是不是？不是。是不是？不是。是不是？找到了！」

書架上有 100 本，你最多翻 100 次。有 1 萬本，你最多翻 1 萬次。資料量多 10 倍，你就慢 10 倍——很公平，成正比。

這種「資料多 10 倍就慢 10 倍」的情況，速度等級叫 **線性時間**。

**情境三：字典式翻找**

你在一本很厚的字典裡找一個字。你不會從第一頁開始翻，你會先翻到中間，看看是太前面還是太後面，然後再切一半，再切一半⋯⋯每次都把範圍砍掉一半。

一本 1000 頁的字典，你大概翻 10 次就能找到目標。100 萬頁的字典？大概也只要 20 次。資料量暴增，但你多做的事很少。

這種「每次砍一半」的情況，速度等級叫 **對數時間**。

**情境四：最慘的狀況**

回到那個沒有編號的書架。這次更慘——你要幫每一本書都找到它的作者資訊，而作者資訊也在另一個沒有編號的書架上。

對於每一本書（100 本），你都要去另一個書架從頭掃一遍（100 本）。100 × 100 = 10,000 次翻找。如果兩邊都有 1000 本呢？1000 × 1000 = 一百萬次。

資料量多 10 倍，速度慢 100 倍。這就是前面「迴圈裡套迴圈」的情況，速度等級叫 **平方時間**。

### 好，現在可以說 Big O 了

工程師會用一個叫 **Big O** 的記號來標記這些速度等級。它就是一個簡寫，告訴你「資料量變大的時候，這段程式大概會慢多快」：

| 速度等級 | Big O 記號 | 白話翻譯 | 生活對應 |
| :--- | :---: | :--- | :--- |
| 常數時間 | $O(1)$ | 不管多少資料，一樣快 | 有編號的書架，直接拿 |
| 對數時間 | $O(\log n)$ | 每次砍一半，超快 | 翻字典 |
| 線性時間 | $O(n)$ | 資料多 10 倍，慢 10 倍 | 從頭掃到尾 |
| 排序等級 | $O(n \log n)$ | 比線性慢一點 | 排序一疊牌 |
| 平方時間 | $O(n^2)$ | 資料多 10 倍，慢 100 倍 | 迴圈套迴圈 |

前端開發最常不小心踩到的就是**平方時間**——一個 `.map()` 裡面呼叫 `.find()`，看起來只是兩行程式碼，但其實是一個迴圈套另一個迴圈。資料量小的時候完全感覺不到，但只要資料從 50 筆變成 5000 筆，畫面就開始卡了。

### 速度等級速查表

你常用的那些陣列方法，速度等級其實很好記：

| 程式碼 | 速度 | 為什麼 |
| :--- | :---: | :--- |
| `map.get('key')` | $O(1)$ | 有索引，直接定位 |
| `set.has(value)` | $O(1)$ | 有索引，直接定位 |
| `arr[5]` | $O(1)$ | 用位置直接拿 |
| `arr.find(x => ...)` | $O(n)$ | 從頭掃到尾 |
| `arr.filter(...)` | $O(n)$ | 每個看一遍 |
| `arr.sort()` | $O(n \log n)$ | 排序就是這個等級 |
| `.map()` 裡面套 `.find()` | $O(n^2)$ | 迴圈套迴圈！危險！ |

有一個常見的誤會：`arr.filter().map().reduce()` 這樣串三個方法，是不是 $n \times n \times n$？不是。它只是把陣列跑了三遍——一遍過濾、一遍轉換、一遍計算——每遍都是從頭到尾走一次，加起來是三倍的 $n$，不是 $n$ 的三次方。真正危險的是**巢狀**：外面的迴圈每跑一圈，裡面又要整個掃一遍。

## 前端開發裡的資料結構

「資料結構」聽起來很學術，但其實就是「資料怎麼擺、怎麼找」。你每天都在用，只是沒有用這個詞去叫它。

### Call Stack：你的程式是怎麼一步步跑的

JavaScript 一次只能做一件事（單執行緒）。那它怎麼知道該先做哪件事？靠的是一個叫 **Call Stack** 的東西——中文叫「呼叫堆疊」。

你可以把 Call Stack 想像成一疊盤子：

- 程式呼叫一個函式 → 在盤子堆的最上面放一個盤子
- 那個函式跑完了 → 把最上面的盤子拿走
- 下面的盤子（之前的函式）才能繼續

最晚放上去的盤子，最先被拿走——這就是 **Stack（堆疊）** 這個資料結構的特性，叫做「後進先出」。

```js
function multiply(a, b) { return a * b }
function square(n) { return multiply(n, n) }
console.log(square(5))
```

這段程式跑的時候，盤子堆的變化是這樣的：

1. `console.log(square(5))` 放上去
2. `square(5)` 被呼叫，放上去
3. `multiply(5, 5)` 被呼叫，放上去（現在有三層）
4. `multiply` 算完回傳 25，拿走
5. `square` 拿到結果，回傳 25，拿走
6. `console.log` 印出 25，拿走（清空了）

你在 DevTools 裡看到的 Error Stack Trace，就是程式出錯那一刻盤子堆的快照——它告訴你「是從哪一路呼叫過來的」。而那個著名的錯誤 `Maximum call stack size exceeded`，就是盤子疊太高了——通常是遞迴沒寫好，函式不斷呼叫自己卻沒有停下來。

### Event Loop：那非同步的事怎麼辦？

JavaScript 一次只能做一件事，那 `setTimeout`、`fetch` 這些要等的東西怎麼辦？總不能整個程式卡在那邊等吧？

靠的是 **Event Loop（事件迴圈）**——它就像一個在 Call Stack 旁邊站崗的管理員。

機制是這樣的：
1. 你寫了一個 `setTimeout(callback, 1000)`，JavaScript 把「等 1 秒」這件事丟給瀏覽器去計時，自己繼續做下面的事
2. 1 秒到了，瀏覽器把那個 callback（你要它做的事）排進一個「等待區」
3. Event Loop 不斷巡邏：「Call Stack 空了嗎？空了！那我從等待區拿一個 callback 來執行。」

這個等待區其實有兩個，而且有優先順序：
- **Microtask Queue**（VIP 等待區）：`Promise.then()` 的 callback 會排在這裡。每次 Call Stack 清空，VIP 等待區的全部 callback 會先被清空。
- **Task Queue**（普通等待區）：`setTimeout`、DOM 事件的 callback 排在這裡。要等 VIP 的全部處理完才輪到它。

這就是為什麼下面這段程式碼的輸出順序不是你直覺想的那樣：

```js
console.log('1')                              // 同步，直接跑
setTimeout(() => console.log('3'), 0)          // 丟進普通等待區
Promise.resolve().then(() => console.log('4')) // 丟進 VIP 等待區
console.log('2')                              // 同步，直接跑

// 輸出：1, 2, 4, 3
// 不是 1, 2, 3, 4！
```

為什麼 `setTimeout(..., 0)` 不是立刻執行？因為就算等 0 秒，它的 callback 也是被排進普通等待區，要等同步程式和 VIP 都跑完才輪到它。Promise 是 VIP，所以 4 先出來，3 最後。

### 前端的百寶箱：Map、Set、Object、WeakMap

這四個東西你多少都用過，它們都是用來「存東西、找東西」的。要理解為什麼它們查找速度快，得先知道一個概念：**Hash Table（雜湊表）**。

#### 為什麼 Hash Table 這麼快？

想像一個有 1000 格的置物櫃。每格都有編號（0 到 999）。

你要存一個叫 `"Jeremy"` 的東西。Hash Table 會先把 `"Jeremy"` 這個字丟進一個「hash function（雜湊函式）」裡——它就像一台魔法機器，吃進去任何東西，吐出一個數字。假設吐出來的是 `42`，那就把東西放進第 42 格。

下次要找 `"Jeremy"` 的時候？再把 `"Jeremy"` 丟進同一台機器，又得到 `42`，直接去第 42 格拿——不用掃 1000 格，一步到位。

這就是為什麼 `Map.get()`、`Set.has()`、`Object` 的屬性存取都是 $O(1)$——它們底層都是用 Hash Table 實作的。

:::info
偶爾會有兩個不同的 key 算出同樣的數字（這叫「碰撞」），這時候需要額外處理，最壞情況下可能退化成 $O(n)$。但在正常使用下，碰撞很少發生，所以我們說 Hash Table 的查找是「平均 $O(1)$」。
:::

#### JS 資料結構 vs 學術資料結構

你在 JavaScript 裡用的這些東西，其實都對應到電腦科學裡正式的資料結構：

| 你在 JS 裡寫的 | 學術上叫做 | 核心特性 |
| :--- | :--- | :--- |
| `Object {}` | Hash Table | key → value 對應，key 限文字，查找 $O(1)$ |
| `Map` | Hash Table | key → value 對應，key 不限型別，查找 $O(1)$ |
| `Set` | Hash Set | 只存值不存對，判斷「在不在」$O(1)$ |
| `Array []` | 動態陣列（Dynamic Array） | 連續記憶體，用 index 存取 $O(1)$，搜尋 $O(n)$ |
| `WeakMap` | Weak Reference Hash Table | 同 Hash Table，但不阻止垃圾回收 |

這個對應關係很重要，因為當你看到有人說「用 Hash Table 優化查找」，其實就是在說「用 `Map` 或 `Set` 取代 `Array.find()`」——你已經會了。

#### 各自的擅長場景

**`Object {}`——最萬用的**

就是你每天都在寫的 `{ name: 'Jeremy', age: 28 }`。key 只能是文字（string），但 V8 對它有大量最佳化，一般場景下超快。

**`Map`——比 Object 更彈性**

key 可以是任何東西（數字、物件、甚至 `NaN` 都行），會記住插入順序，頻繁新增刪除時效能更穩定。當你需要用「不是文字」的東西當 key，`Map` 比 `Object` 合適。

**`Set`——只管「有沒有」**

不存鍵值對，只存一堆不重複的值。核心能力就是回答：「這個東西在不在裡面？」而且靠 Hash Table 直接定位，不管 Set 有多大都一樣快。前面的去重就是 `Set` 的經典用法。

**`WeakMap`——會自動清垃圾的 Map**

跟 `Map` 很像，但 key 一定要是物件。特殊的地方在於：如果那個物件在其他地方都沒人用了，`WeakMap` 裡的對應條目會自動被清掉，不會佔記憶體。Vue 3 的響應式系統內部就大量使用 `WeakMap` 來追蹤哪些物件是 reactive 的。

| 東西 | 底層結構 | key 限制 | 查找速度 | 什麼時候用 |
| :--- | :--- | :--- | :---: | :--- |
| `Object {}` | Hash Table | 文字 | $O(1)$ | 一般場景 |
| `Map` | Hash Table | 不限 | $O(1)$ | key 不是文字、頻繁增刪 |
| `Set` | Hash Set | —（只存值） | $O(1)$ | 去重、判斷有沒有 |
| `WeakMap` | Weak Hash Table | 物件 | $O(1)$ | 避免記憶體洩漏 |
| `Array []` | 動態陣列 | — | $O(1)$ 索引 / $O(n)$ 搜尋 | 有序資料、需要 index |

## 寫 Vue 的時候，藏在程式碼裡的速度差異

這段不是要講 Vue 內部什麼 Virtual DOM diff 演算法——那是框架開發者要操心的事。我想聊的是：我們平常寫 Vue 的時候，那些看起來很正常的寫法，背後其實有快有慢。

### 例子一：computed 裡的隱形陷阱

假設你在做一個商品列表，每個商品有一個分類 ID，你想把分類名稱補上去：

```vue
<script setup>
import { computed } from 'vue'

const props = defineProps({
  items: Array,       // [{id: 1, name: '蘋果', categoryId: 'fruit'}, ...]
  categories: Array,  // [{id: 'fruit', name: '水果'}, {id: 'meat', name: '肉類'}, ...]
})

// 直覺寫法
const enrichedItems = computed(() =>
  props.items.map(item => {
    // 對每一個商品，都跑進 categories 裡面從頭到尾找一遍
    const cat = props.categories.find(c => c.id === item.categoryId)
    return { ...item, categoryName: cat?.name }
  })
)
</script>
```

看起來很乾淨對吧？`.map()` 裡面用 `.find()` 查分類名稱，邏輯很清楚。

但你有沒有意識到：**對每一個商品，`.find()` 都要把 categories 從頭掃到尾**？如果有 500 個商品、100 個分類，每次 computed 重算就是 500 × 100 = 五萬次比較。而且只要 `items` 一變動（比如搜尋結果改變），整個 computed 就會重跑一次。

改成先建一個「索引簿」（也就是 Hash Table）：

```vue
<script setup>
import { computed } from 'vue'

const props = defineProps({
  items: Array,
  categories: Array,
})

// 先把 categories 建成一個 Map（索引簿）
// key 是分類 ID，value 是分類物件
// 這一步跑一遍 categories 就做完了
const categoryMap = computed(() =>
  new Map(props.categories.map(c => [c.id, c]))
)

// 現在查分類名稱不用掃了，直接用 ID 去 Map 裡面撈
const enrichedItems = computed(() =>
  props.items.map(item => ({
    ...item,
    categoryName: categoryMap.value.get(item.categoryId)?.name
  }))
)
</script>
```

`categoryMap` 只有在 `categories` 變動時才重建（Vue 的 `computed` 有快取功能），而每次查找都是直接定位，不用掃。整體從「迴圈套迴圈」降級成「掃一遍就結束」——快很多。

### 例子二：v-for 裡面硬塞 v-if

另一個常見的場景：

```vue
<template>
  <!-- 不太好的寫法 -->
  <div v-for="item in items" :key="item.id">
    <span v-if="item.active">{{ item.name }}</span>
  </div>
</template>
```

這樣寫，Vue 會把 `items` 裡面每一筆都跑過 `v-for`，每一筆都生成 DOM 節點，然後才用 `v-if` 決定要不要顯示。就算 1000 筆裡面只有 10 筆是 `active` 的，Vue 還是得跑 1000 圈。

比較好的做法：先過濾好再丟給 `v-for`。

```vue
<script setup>
import { computed } from 'vue'

// 先把 active 的篩出來，交給 computed 快取
const activeItems = computed(() =>
  props.items.filter(item => item.active)
)
</script>

<template>
  <!-- 只跑 active 的那些，DOM 節點也少很多 -->
  <div v-for="item in activeItems" :key="item.id">
    <span>{{ item.name }}</span>
  </div>
</template>
```

### 但是——不要為了效能犧牲一切

講了這些效能差異，我覺得有一件事很重要，而且是很多文章不會跟你說的：

**資料量小的時候，好讀比快更重要。**

```js
// 這個超好讀，一眼就知道在幹嘛
const userName = users.find(u => u.id === currentId)?.name

// 這個比較快，但你得多想一秒才能理解
const userMap = new Map(users.map(u => [u.id, u]))
const userName = userMap.get(currentId)?.name
```

如果 `users` 只有 20 個人而且不會增長，用 `.find()` 就好了啊。為了 20 筆資料去建一個 `Map`，其實是在增加程式碼的複雜度，換來的效能提升你根本感覺不到。

我自己的判斷方式是：

- **資料很少（幾十筆）、不會成長**：怎麼好讀怎麼寫，不需要想太多
- **資料會成長、或是在 `computed` 裡頻繁被重算**：先建個 `Map` 或 `Set` 做索引
- **不確定的時候**：先寫好讀的版本。等哪天真的卡了，打開 DevTools 的 Performance 面板看數據，有證據再優化

「過早優化是萬惡之源」這句話是 Donald Knuth（電腦科學的大佬）說的。他不是叫你不要優化，是叫你不要在沒有證據的情況下瞎猜哪裡慢。先讓程式碼能跑、好讀、好維護，速度的問題等到真的出現再處理。

## 前端跟資料庫有什麼關係？

這個段落可能會讓你意外——前端工程師幹嘛要知道資料庫的事？

但你仔細想想，前端其實每天都在做「資料庫」做的事，只是規模小一點、跑在瀏覽器裡而已。

### 你的 Pinia Store 就是一個迷你資料庫

你在 Pinia 裡寫的 store，其實就是一個跑在瀏覽器記憶體裡的小型資料庫：

```js
// stores/product.js
export const useProductStore = defineStore('product', () => {
  const products = ref([])          // 這就是你的「資料表」
  const loading = ref(false)

  async function fetchAll() {       // 「把全部資料撈出來」
    loading.value = true
    products.value = await api.getProducts()
    loading.value = false
  }

  function getById(id) {            // 「用 ID 查某一筆」
    return products.value.find(p => p.id === id)
  }

  return { products, loading, fetchAll, getById }
})
```

存資料、查資料、更新資料——這不就是資料庫在做的事嗎？差別只在於你的「資料庫」跑在瀏覽器裡，重新整理就沒了。

### 正規化：把資料攤平，改一處就好

在資料庫的世界裡，有一個很重要的概念叫**正規化（Normalization）**。講白話就是：**不要讓同一份資料重複出現在好幾個地方。**

為什麼？因為資料一重複，你就會遇到這些問題：
- **改了一邊忘了另一邊**：Jeremy 的名字在三篇文章裡各存了一份，改了其中一篇，另外兩篇還是舊的
- **資料不一致**：同一個人在不同地方顯示不同的名字，使用者會覺得是 bug
- **浪費空間**：同樣的東西存了好幾份

前端的 store 也會遇到一模一樣的問題。假設 API 回傳的資料長這樣：

```json
[
  { "id": 1, "title": "文章一", "author": { "id": 10, "name": "Jeremy" } },
  { "id": 2, "title": "文章二", "author": { "id": 10, "name": "Jeremy" } },
  { "id": 3, "title": "文章三", "author": { "id": 20, "name": "Aira" } }
]
```

Jeremy 的資料重複出現了兩次。如果你直接把這包 JSON 存進 store，就是「未正規化」的狀態。

正規化的做法是**把巢狀資料攤平**——每種東西各存一份，用 ID 去關聯：

```js
// 未正規化：巢狀結構，同一個作者重複存了好幾份
const denormalized = {
  posts: [
    { id: 1, title: '文章一', author: { id: 10, name: 'Jeremy' } },
    { id: 2, title: '文章二', author: { id: 10, name: 'Jeremy' } },  // Jeremy 又出現了
    { id: 3, title: '文章三', author: { id: 20, name: 'Aira' } },
  ]
}

// 正規化：攤平成 { [id]: entity } 的結構
const normalized = {
  posts: {
    1: { id: 1, title: '文章一', authorId: 10 },
    2: { id: 2, title: '文章二', authorId: 10 },
    3: { id: 3, title: '文章三', authorId: 20 },
  },
  authors: {
    10: { id: 10, name: 'Jeremy' },   // Jeremy 只存一份
    20: { id: 20, name: 'Aira' },     // Aira 也只存一份
  }
}
```

正規化之後：

- **改名只改一處**：`normalized.authors[10].name = 'Jeremy Ho'`，所有引用 Jeremy 的文章自動拿到新名字
- **查找超快**：`normalized.posts[2]` 直接用 ID 拿，是 $O(1)$，不用 `.find()`
- **資料不重複**：每個作者只存一份，省空間也避免不一致

把陣列轉成 `{ [id]: entity }` 的格式，其實就是在前端手動建了一個 Hash Table——key 是 ID，value 是那筆資料。跟前面說的「先建 Map 再查」是完全一樣的思路。

Redux 社群早期就大力推廣這種 normalized state 的模式，Vue 生態系裡雖然比較少強調，但邏輯是一樣的。

不過實務上，小型專案不一定要做到這個程度。如果你的資料結構不複雜、資料量不大、作者不會改名，直接存 API 回傳的格式也沒問題。這又回到那個平衡：好讀好維護 vs 理論上更「正確」。

### 瀏覽器裡的「真正」資料庫

有時候你需要資料在關掉瀏覽器後還留著——不是那種重新整理就消失的 store。

**localStorage**：最簡單。就是一個你可以存東西的小倉庫，key-value 格式，容量大約 5-10 MB。適合存使用者偏好設定（深色模式、語系選擇），不適合存大量資料。

**IndexedDB**：瀏覽器內建的完整資料庫。容量大（幾百 MB 以上）、可以建索引加速查詢、支援交易機制——名字裡的 "Indexed" 就是說它可以像真正的資料庫一樣建索引。PWA（那種安裝在手機上的網頁 app）能離線運作，靠的就是把資料存在 IndexedDB 裡。

### 前端的快取，就是一種資料庫策略

你的前端 store 裡面存的資料，其實是後端資料庫的「副本」。後端資料庫是正本，你的 store 是影本。

問題來了：影本什麼時候會過期？要不要每次都重新跟後端要？

有一種常見的策略叫 **Stale-While-Revalidate**（先用舊的，背景偷偷更新）：

1. 使用者打開頁面，先用 store 裡面快取的資料渲染畫面（就算可能過期也沒關係）
2. 同時在背景發一個 API 去拿最新的資料
3. 拿到後靜靜地更新畫面

使用者不用看到白屏等待，體驗好很多。TanStack Query（Vue Query）這類工具幫你做的就是這件事——管理快取、判斷資料有沒有過期、背景自動更新。你寫的 `useQuery()` 其實就是在做這整套流程。

還有一種叫**樂觀更新**：使用者按下「按讚」，不等 API 回應就先把讚的數字 +1。如果 API 成功了，什麼都不用做；如果失敗了，再把數字 -1 改回來。先假設會成功，失敗再回滾——這跟資料庫裡「交易失敗就回滾」是同一個概念。

### API 打太多次的問題

最後講一個後端工程師很愛提的概念，但前端也會踩到。

假設你要渲染一個文章列表，每篇文章要顯示作者名稱。如果你對每篇文章都打一次 API 去撈作者：

```js
const posts = await fetchPosts()        // 第 1 次 API
// 然後對每篇文章各打一次
for (const post of posts) {
  post.author = await fetchAuthor(post.authorId)  // 第 2, 3, 4, 5... 次 API
}
```

10 篇文章就打 11 次 API。100 篇就 101 次。每次 API 都有網路延遲，加起來超慢。

更聰明的做法：一次把所有需要的作者批次撈回來，然後在前端用 `Map` 配對。

```js
const posts = await fetchPosts()
// 把所有不重複的作者 ID 收集起來
const authorIds = [...new Set(posts.map(p => p.authorId))]
// 一次撈回所有作者
const authors = await fetchAuthorsByIds(authorIds)
// 建個索引簿，配對就秒完
const authorMap = new Map(authors.map(a => [a.id, a]))

const postsWithAuthors = posts.map(post => ({
  ...post,
  author: authorMap.get(post.authorId)
}))
```

2 次 API 搞定，前端用 `Map` 做配對。又是「先建索引再查」的老招。

## 串起來

回頭看這一整篇，其實說的都是同一件事：**資料怎麼擺，決定了你的程式有多快。**

- 用 `Set` 取代 `includes`，不用再從頭掃到尾
- 先建 `Map` 再查，不用每次都整個翻一遍
- Store 裡的資料做攤平，改一個地方就好
- API 做批次處理，不要一筆一筆慢慢撈

你不需要會手刻紅黑樹才算「懂演算法」。在前端的世界裡，演算法思維通常就是在問一個很簡單的問題：

**「如果資料量變成 10 倍，這段程式還撐得住嗎？」**

能回答這個問題，你就已經在用演算法了。
