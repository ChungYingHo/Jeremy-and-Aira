## 漸進符號的直覺

「漸進」這個詞聽起來有點抽象，但概念很直觀：當輸入規模 $n$ 夠大的時候，函數裡的低階項跟常數係數根本無關緊要，只有最高階項的形狀才重要。

舉個例子：$T(n) = 3n^2 + 100n + 5000$

當 $n = 10000$ 時：
- $3n^2 = 3 \times 10^8$
- $100n = 10^6$（只有 $n^2$ 項的 0.3%）
- $5000$（根本可以忽略不計）

所以我們說這個函數是 $O(n^2)$，因為在 $n$ 夠大的時候，$n^2$ 這項完全主導了整體的增長。

化簡規則：

1. 捨棄常數係數：$O(5n) = O(n)$
2. 捨棄低階項：$O(n^2 + n) = O(n^2)$
3. 不同輸入各自獨立：$O(n + m)$ 不能化簡成 $O(n)$，因為 $m$ 可能比 $n$ 大

## O、Ω、Θ 的差異

三個符號在直覺上可以這樣理解：

- $O$（Big-O）：**上界**，保證演算法不會比這更慢。可以想成「最壞情況最多跑這麼久」。
- $\Omega$（Big-Omega）：**下界**，保證演算法不會比這更快。也就是「再怎麼幸運也至少要跑這麼久」。
- $\Theta$（Big-Theta）：**緊界**，上界和下界都被卡住，成長速度就是這個等級。

<AsymptoticBoundsDiagram client:visible />

圖裡的三條線說的就是這件事：在 $n_0$ 之後，$f(n)$ 被夾在 $c' \cdot g(n)$（下界）和 $c \cdot g(n)$（上界）之間。這就是 $\Theta$ 成立的條件——存在某個 $n_0$ 和兩個正常數 $c, c'$，使得對所有 $n \geq n_0$：

$$c' \cdot g(n) \leq f(n) \leq c \cdot g(n)$$

用線性搜尋來具體說明三者的差異：

- **Worst case = $O(n)$**：目標在最後一個位置，要掃完整個陣列
- **Best case = $\Omega(1)$**：目標在第一個位置，看一次就找到了
- **Average case = $\Theta(n)$**：期望上要跑一半，仍然是線性成長

:::info
業界日常說的 Big O 幾乎都是指 worst case 的上界，這是一個習慣上的簡化。但在嚴格的學術討論裡，$O$、$\Omega$、$\Theta$ 各有明確的定義，說「Quick Sort 是 $O(n \log n)$」是不精確的（因為它的 worst case 是 $O(n^2)$），說「$\Theta(n \log n)$」才代表緊界。這個差異在讀論文或跟人討論演算法的時候值得注意。
:::

## 常見複雜度等級（深入）

這節不只列等級，而是說清楚每個等級的特性跟它背後的直覺。

### 常數時間 O(1)

不論輸入多大，執行時間固定。這代表演算法完全不依賴輸入規模，做的是固定幾個操作。典型範例是陣列 index 存取（直接算記憶體位址）、hash table 在沒有碰撞的理想情況下的查找。

### 對數時間 O(log n)

每次把問題規模砍一半（或更大的比例）。二分搜尋法是最典型的例子：從 $n$ 個元素開始，每次排除一半，剩下一個時才停止，共砍 $\log_2 n$ 次，所以是 $O(\log n)$。對數成長非常緩慢——$n$ 從 100 萬到 10 億，$\log_2 n$ 只從 20 增加到 30。

### 線性時間 O(n)

跟輸入規模成正比，要把每個元素至少看一遍。線性搜尋、找最大值、計算陣列總和都是這個等級。這通常是「問題本身的下界」——你至少要讀過每個輸入一次，所以很多問題天然就是 $\Omega(n)$。

### 線性對數時間 O(n log n)

通常出現在「對每個元素做一件 $\log n$ 代價的事」的結構，或者「分而治之，每層合併需要線性代價」。Merge Sort 和 Heap Sort 都是這個等級。這也是比較排序（comparison sort）的理論下界——任何只靠比較來排序的演算法，最壞情況不可能比 $\Omega(n \log n)$ 更快，這在資訊理論上有嚴格的證明。

### 多項式時間 O(nᵏ)

$n$ 的某個固定次方。最常見的是 $O(n^2)$ 和 $O(n^3)$。

$O(n^2)$ 通常來自兩層巢狀迴圈，對每個元素都要跑一遍所有其他元素——Bubble Sort、Insertion Sort、Selection Sort 都是。$n = 1000$ 時大約 $10^6$ 次操作，還可以接受；$n = 10^5$ 時就是 $10^{10}$ 次，在現代硬體上大概要幾十秒，很慢了。

$O(n^3)$ 常見於三層巢狀迴圈，標準的矩陣乘法就是 $O(n^3)$：兩個 $n \times n$ 的矩陣相乘，每個輸出元素要做 $n$ 次乘加，共 $n^2$ 個輸出元素。$n = 1000$ 的矩陣乘法需要 $10^9$ 次操作，需要幾秒鐘。

只要 $k$ 是固定常數，多項式時間演算法在理論上都被認為是「可接受的」——這也是 P 類問題的定義（詳見最後一節）。

### 指數時間 O(2ⁿ)

通常來自「列舉所有子集合」，因為一個有 $n$ 個元素的集合共有 $2^n$ 個子集。暴力解背包問題、子集合加總問題都是這個等級。$n = 30$ 時已經是 $10^9$ 次，$n = 50$ 就完全無法實用了。

### 階乘時間 O(n!)

列舉所有排列的代價。暴力解旅行推銷員問題要列出所有城市的排列，共 $n!$ 種。$n = 12$ 時大約 $4.8 \times 10^8$，勉強可以跑；$n = 15$ 就超過 $10^{12}$，不可行了。

## 最壞、最佳、平均情況

同一個演算法，面對不同的輸入，執行時間可能差很多。這就是為什麼我們需要區分三種情況：

- **Worst case**：輸入對演算法最不利的情況。這是 $O$ 最常對應的場景。
- **Best case**：輸入對演算法最有利的情況。這是 $\Omega$ 最常對應的場景。
- **Average case**：對所有可能輸入取平均（需要假設輸入分佈）。

Quick Sort 是說明這三種情況最有代表性的例子。它的核心是選一個 pivot，把陣列分成兩部分，然後遞迴排序。

- **Best / Average case：$O(n \log n)$**：每次 pivot 大概把陣列切成一半，遞迴深度是 $\log n$，每層合計掃 $n$ 個元素，共 $n \log n$。
- **Worst case：$O(n^2)$**：每次 pivot 都選到最大或最小值，陣列被切成 1 和 n-1，退化成一層一層剝，遞迴深度變 $n$，共 $n^2$。這在已排序的輸入上很容易發生（取決於 pivot 選法）。

| 演算法 | Best | Average | Worst |
| :--- | :--- | :--- | :--- |
| Binary Search | $O(1)$ | $O(\log n)$ | $O(\log n)$ |
| Linear Search | $O(1)$ | $O(n)$ | $O(n)$ |
| Bubble Sort | $O(n)$ | $O(n^2)$ | $O(n^2)$ |
| Merge Sort | $O(n \log n)$ | $O(n \log n)$ | $O(n \log n)$ |
| Quick Sort | $O(n \log n)$ | $O(n \log n)$ | $O(n^2)$ |

:::info
Merge Sort 的三種情況全都是 $\Theta(n \log n)$，這讓它在輸入分佈未知時是更穩定的選擇。Quick Sort 在實作上通常比 Merge Sort 快（cache 友好、原地排序），但需要好的 pivot 策略（如隨機選 pivot 或 median-of-three）來避免 worst case。Merge Sort 的代價是需要 $O(n)$ 的額外空間來合併。
:::

## 主定理（Master Theorem）

分治法是一類非常常見的演算法設計策略：把問題拆成若干個較小的子問題，遞迴解決，再把結果合併。這類演算法的時間複雜度通常長這樣：

$$T(n) = aT\!\left(\frac{n}{b}\right) + f(n), \quad a \geq 1,\; b > 1$$

- $a$：每次遞迴拆成幾個子問題
- $b$：每個子問題的規模縮小比例（變成 $n/b$）
- $f(n)$：每次遞迴的合併/拆分代價

光靠這個式子，很難直接看出 $T(n)$ 的成長速度。主定理給了我們一個直接的方法。

**定義關鍵值** $c = \log_b a$

這個 $c$ 代表什麼？若暫時忽略合併代價，整棵遞迴樹展開到底共有 $n^c$ 個葉節點，所以 $n^c$ 是「純遞迴展開」的天然代價。主定理的核心就是比較 $f(n)$ 跟 $n^c$，看誰主導整體複雜度。

<MasterTheoremDiagram client:visible />

### Case 1：遞迴主導

$f(n)$ 比 $n^c$ 成長慢（差了一個多項式因子），代表合併代價相對小，遞迴展開的葉節點代價主導整體。

$$T(n) = \Theta(n^c)$$

**範例**：$T(n) = 8T(n/2) + O(n^2)$

$a = 8,\; b = 2$，所以 $c = \log_2 8 = 3$。$f(n) = O(n^2) = O(n^{3-1})$，滿足 Case 1（$\varepsilon = 1$），因此 $T(n) = \Theta(n^3)$。

### Case 2：勢均力敵，加 log

$f(n)$ 跟 $n^c$ 同個量級，遞迴和合併的代價相當，每一層都貢獻類似的量，共 $\log n$ 層，所以乘上一個 $\log n$。

$$T(n) = \Theta(n^c \log n)$$

**範例**：Merge Sort，$T(n) = 2T(n/2) + O(n)$

$a = 2,\; b = 2$，所以 $c = \log_2 2 = 1$。$f(n) = O(n) = \Theta(n^1)$，滿足 Case 2，因此 $T(n) = \Theta(n \log n)$。

### Case 3：合併主導

$f(n)$ 比 $n^c$ 成長快（且還需滿足正則條件），代表合併代價在每層遞迴中越來越重，頂層的合併代價主導整體。

$$T(n) = \Theta(f(n))$$

**範例**：$T(n) = T(n/2) + O(n)$

$a = 1,\; b = 2$，所以 $c = \log_2 1 = 0$。$f(n) = O(n) = \Omega(n^{0+1})$，滿足 Case 3（$\varepsilon = 1$），因此 $T(n) = \Theta(n)$。

:::tip
記憶主定理的口訣：**$n^c$ 是分治的天然代價**。比它慢的合併讓遞迴葉節點決定（Case 1）；跟它差不多的合併加一個 log（Case 2）；比它快的合併讓合併本身決定（Case 3）。
:::

:::warning
主定理不是萬能的，有些遞迴式不在它的適用範圍內。例如 $T(n) = 2T(n/2) + n\log n$，這裡 $f(n) = n\log n$，它比 $n^c = n^1$ 大，但差的只是一個 $\log n$ 因子，而不是多項式因子 $n^\varepsilon$，所以不滿足 Case 3 的條件，也不屬於 Case 2，主定理無法直接套用。這類情況要改用遞迴樹法（recursion tree）或代入法（substitution method）來分析，結果是 $T(n) = \Theta(n \log^2 n)$。
:::

## 問題複雜度（簡介）

到目前為止講的複雜度都是「這個演算法有多快」。但還有另一個維度值得知道：**問題本身有多難**，也就是解這個問題的最快演算法最少需要多快。

**P 問題**

能在多項式時間（$O(n^k)$ 以內）內「解決」的問題集合。排序、最短路徑（Dijkstra）、最大流都是 P 問題。

**NP 問題**

能在多項式時間內「驗證」一個解是否正確的問題集合。注意「驗證」跟「找到解」不一樣。給你一個背包問題的解（哪些物品要拿），你可以快速算出總重和總價值、確認對不對。但要找到最佳解，目前沒有已知的多項式時間演算法。

所有 P 問題都是 NP 問題（能解的一定能驗證），但 NP 問題是否都是 P 問題，目前不知道。

**NP-complete**

NP 裡最難的一批問題。這些問題有一個特性：所有 NP 問題都可以在多項式時間內「規約」到它們。這代表如果你能找到一個 NP-complete 問題的多項式時間演算法，你就同時解決了所有 NP 問題。旅行推銷員問題（TSP）、布林可滿足性問題（SAT）都是 NP-complete。

**P = NP？**

這是電腦科學最大的未解問題。如果 P = NP，代表所有能快速驗證的問題都能快速解決，現代密碼學的基礎（依賴「驗證容易、解決難」的問題）會直接崩潰。目前幾乎所有人相信 P ≠ NP，但這還沒有被嚴格證明，Clay 數學研究所把它列為七個千禧年大獎問題之一。

:::info
P vs NP 不影響日常演算法的學習，但理解這個框架有一個很實用的價值：如果你面對的問題被證明是 NP-complete，那就代表你可以放棄尋找精確的多項式時間解，改走近似演算法（保證解的品質在最優解的某個比例內）或啟發式搜尋（如基因演算法、模擬退火），這樣才是務實的策略。
:::
