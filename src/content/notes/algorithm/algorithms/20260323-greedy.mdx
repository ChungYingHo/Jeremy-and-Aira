## 什麼是貪婪演算法

「貪婪」這個名字取得很傳神。貪婪演算法（Greedy Algorithm）的核心思想只有一句話：**每一步都選當下看起來最好的選項，不管之後會怎樣。**

舉個日常的例子。你去便利商店買了 73 元的東西，付了 100 元，店員要找你 27 元。店員腦中跑的其實就是一個貪婪演算法：

1. 還要找 27 元，最大面額不超過 27 的硬幣是 10 元 → 拿一枚，剩 17 元
2. 還要找 17 元，最大面額不超過 17 的硬幣是 10 元 → 拿一枚，剩 7 元
3. 還要找 7 元，最大面額不超過 7 的硬幣是 5 元 → 拿一枚，剩 2 元
4. 還要找 2 元，拿兩枚 1 元

結果：10 + 10 + 5 + 1 + 1 = 27 元，用了 5 枚硬幣，而且這就是最少枚數。

不回頭、不悔棋、每步都取當下最大——這就是貪婪。

:::warning
這個策略之所以對台灣硬幣有效，是因為台灣的面額設計（1、5、10、50、100、500）恰好滿足貪婪選擇性質。換一套面額就不一定了——如果硬幣是 **{1, 3, 4}**，要找 **6 元**，貪婪會選 4+1+1 共 3 枚，但最優解是 3+3 共 2 枚。一般的找零問題（任意面額）要用動態規劃才能保證正確。
:::

## 貪婪成立的兩個條件

貪婪演算法並不是萬能的。它之所以在某些問題上能得到正確答案，是因為那些問題具備兩個特性：

**貪婪選擇性質（Greedy Choice Property）**

局部最優的選擇，最終能組合成全域最優解。也就是說，每一步的貪婪選擇不會讓後面的最優解消失。

**最優子結構（Optimal Substructure）**

問題的最優解，包含了子問題的最優解。這跟動態規劃的要求一樣——整體最優，是由局部最優拼起來的。

這兩個性質缺一不可。如果問題具備這兩個特性，貪婪演算法通常能在很低的複雜度下得到正確答案。如果沒有，貪婪很可能給出錯誤結果。

:::info
判斷一個問題能不能用貪婪，通常需要數學證明（反證法或交換論證）。在競程或工作中，如果你沒辦法證明貪婪是對的，先試著找反例——能找到就代表得換方法。
:::

## 圖的基本概念

這一節先把圖論的基礎術語交代清楚，後面三個演算法（Prim、Kruskal、Dijkstra）都以圖作為輸入。

### 無向圖與有向圖

**無向圖（Undirected Graph）** 的邊沒有方向，A 連到 B 就等於 B 連到 A。你可以把它想像成雙向道路，或者朋友關係（你是我朋友，我也是你朋友）。

**有向圖（Directed Graph / Digraph）** 的邊有方向，A → B 不代表 B → A。單行道、Twitter 的追蹤關係（你追蹤我，但我不一定追蹤你）都是有向圖。

**加權圖（Weighted Graph）** 在每條邊上附加一個數值（權重），代表距離、成本、時間等。Prim、Kruskal、Dijkstra 處理的都是加權圖。

### 樹是什麼

**樹（Tree）** 在圖論裡的定義是：連通、無環的無向圖。有 $V$ 個節點的樹恰好有 $V - 1$ 條邊。

樹有一個很重要的性質：**任意兩個節點之間的路徑唯一**。不像一般圖可能有很多條路可以走，樹裡從 A 走到 B 只有一條路。

這讓樹路徑的問題比一般圖簡單很多——不需要 Dijkstra，直接用 BFS 或 DFS 找到那條唯一的路徑就行了。如果是加權樹，路徑權重就是沿途邊的權重加總。

<WeightedTreeDiagram client:visible />

### 路徑、距離、最短路徑

- **路徑（Path）**：從一個節點走到另一個節點，經過的節點與邊的序列
- **距離（Distance）**：路徑上所有邊的權重總和（未加權圖通常以邊數計）
- **最短路徑（Shortest Path）**：所有可能路徑中距離最小的那條

對無向圖和有向圖來說，最短路徑的演算法不同：
- **無向圖**：Dijkstra（非負權重）、Bellman-Ford（可負權重）
- **有向圖（DAG）**：可以用拓撲排序優化，或同樣用 Dijkstra / Bellman-Ford
- **樹**：路徑唯一，不需要最短路徑演算法

## 最小生成樹

在繼續講 Prim 和 Kruskal 之前，先說清楚它們要解的問題是什麼。

**最小生成樹（Minimum Spanning Tree，MST）**：在一個連通加權無向圖中，找出一棵「生成樹」，使得所有邊的權重總和最小。

「生成樹」的意思是：包含圖中所有節點、且構成一棵樹（連通、無環、$V - 1$ 條邊）的子圖。

現實的例子：你有好幾個城市要連接電纜，每對城市之間鋪電纜的成本不同。你希望所有城市都能互通，但總成本要最低——這就是最小生成樹問題。

### Prim 演算法

Prim 的策略：**從任意節點出發，每次把「最便宜的、能把一個新節點拉進來的邊」加入生成樹，直到所有節點都在樹裡。**

你可以把它想成一棵樹在「長大」：從一個節點開始，不斷把最近的新節點吸收進來。

以上面的圖為例，從節點 A 出發：

1. 目前在樹裡：\{A\}。可用邊：A-B(4)、A-D(1)、A-E(8)。選最小 A-D(1)。
2. 目前在樹裡：\{A, D\}。可用邊：A-B(4)、A-E(8)、D-E(7)。選最小 A-B(4)。
3. 目前在樹裡：\{A, D, B\}。可用邊：A-E(8)、D-E(7)、B-E(2)、B-C(3)。選最小 B-E(2)。
4. 目前在樹裡：\{A, D, B, E\}。可用邊：A-E(已連通，跳過)、D-E(已連通，跳過)、B-C(3)、E-C(5)、E-F(9)。選最小 B-C(3)。
5. 目前在樹裡：\{A, D, B, E, C\}。可用邊：C-F(6)、E-F(9)。選最小 C-F(6)。
6. 所有節點都在樹裡，完成。總權重：1+4+2+3+6 = **16**。

```python
import heapq

# graph = {'A': [('B', 4), ('D', 1)], 'B': [('A', 4), ...], ...}
def prim(graph, start):
    in_tree = set()
    heap = [(0, start)]   # (邊的權重, 節點)
    total = 0

    while heap:
        weight, node = heapq.heappop(heap)
        if node in in_tree:
            continue      # 已在 MST 裡，跳過
        in_tree.add(node)
        total += weight

        # 把這個節點的所有鄰居邊加進候選池
        for neighbor, w in graph[node]:
            if neighbor not in in_tree:
                heapq.heappush(heap, (w, neighbor))

    return total
```

時間複雜度：使用優先佇列（min-heap）時為 $O(E \log V)$，其中 $E$ 是邊數、$V$ 是節點數。

### Kruskal 演算法

Kruskal 的策略：**把所有邊按權重由小到大排序，依序加入邊，只要加入這條邊不會形成環，就加入；否則跳過。**

跟 Prim 不同，Kruskal 不從某個節點出發，而是「全域」地從最小邊開始挑，讓各個節點集合慢慢合併。

要判斷「加入這條邊會不會成環」，用的是一個叫做 **Union-Find（並查集）** 的資料結構。它能快速回答「這兩個節點是否已經在同一棵子樹裡」，如果是，加這條邊就會成環；如果不是，就可以合併。

同樣以上面的圖為例：

所有邊排序：A-D(1)、B-E(2)、B-C(3)、A-B(4)、E-C(5)、C-F(6)、D-E(7)、A-E(8)、E-F(9)

1. A-D(1)：A 和 D 不同集合，加入。集合：\{A,D\}, \{B\}, \{C\}, \{E\}, \{F\}
2. B-E(2)：B 和 E 不同集合，加入。集合：\{A,D\}, \{B,E\}, \{C\}, \{F\}
3. B-C(3)：B 和 C 不同集合，加入。集合：\{A,D\}, \{B,E,C\}, \{F\}
4. A-B(4)：A 和 B 不同集合，加入。集合：\{A,D,B,E,C\}, \{F\}
5. E-C(5)：E 和 C 已在同一集合，跳過（會成環）
6. C-F(6)：C 和 F 不同集合，加入。集合：\{A,D,B,E,C,F\}
7. 所有節點都在同一集合，完成。總權重：1+2+3+4+6 = **16**。

```python
# Union-Find：追蹤每個節點屬於哪個集合
parent = {}

def find(x):
    # 往上找到這個集合的根節點
    while parent[x] != x:
        parent[x] = parent[parent[x]]  # 路徑壓縮，讓樹更扁
        x = parent[x]
    return x

def union(x, y):
    rx, ry = find(x), find(y)
    if rx == ry:
        return False   # 已在同一集合 → 加入會成環
    parent[ry] = rx    # 合併兩個集合
    return True

# edges = [('A', 'D', 1), ('B', 'E', 2), ...]
def kruskal(nodes, edges):
    for n in nodes:
        parent[n] = n              # 初始化：每個節點自成一集合

    edges.sort(key=lambda e: e[2]) # 按權重由小到大排序
    total = 0

    for u, v, w in edges:
        if union(u, v):            # 不成環才納入 MST
            total += w

    return total
```

時間複雜度：$O(E \log E)$，主要來自排序。

### Prim vs Kruskal

兩者都是貪婪演算法，都能找到最小生成樹，結果可能不同（有多棵 MST 時），但總權重一定相同。

| | Prim | Kruskal |
| :--- | :--- | :--- |
| 方向 | 從節點出發，逐步擴張 | 從邊出發，合併集合 |
| 適合 | 稠密圖（邊多） | 稀疏圖（邊少） |
| 需要 | 優先佇列 | Union-Find + 排序 |
| 複雜度 | $O(E \log V)$ | $O(E \log E)$ |

## Dijkstra 最短路徑

Dijkstra 演算法解的問題是：給定一個加權圖（有向或無向），從起點出發，找到到達所有其他節點的最短路徑。

**貪婪策略**：每次從「尚未確定最短距離的節點」中，選出目前距離起點最近的節點，把它標記為「已確定」，然後透過它更新鄰居的距離。

這個「每次選最近的」就是貪婪。它之所以正確，是因為一旦某個節點被選出來，從起點到它的距離就已經是最短了——不可能透過其他節點繞更遠的路反而更近（前提是沒有負權重邊）。

以這個有向加權圖為例，從 A 出發：

<DijkstraDiagram client:visible />

| 步驟 | 已確定 | dist[A] | dist[B] | dist[C] | dist[D] | dist[E] |
| :---: | :--- | :---: | :---: | :---: | :---: | :---: |
| 初始 | {} | 0 | ∞ | ∞ | ∞ | ∞ |
| 選 A(0) | \{A\} | 0 | 2 | 4 | ∞ | ∞ |
| 選 B(2) | \{A,B\} | 0 | 2 | 4 | 5 | ∞ |
| 選 C(4) | \{A,B,C\} | 0 | 2 | 4 | 5 | 9 |
| 選 D(5) | \{A,B,C,D\} | 0 | 2 | 4 | 5 | 9 |
| 選 E(9) | \{A,B,C,D,E\} | 0 | 2 | 4 | 5 | 9 |

```python
import heapq

# graph = {'A': [('B', 2), ('C', 4)], ...}
def dijkstra(graph, start):
    dist = {node: float('inf') for node in graph}
    dist[start] = 0          # 起點到自己距離為 0
    heap = [(0, start)]      # (目前距離, 節點)

    while heap:
        d, node = heapq.heappop(heap)
        if d > dist[node]:
            continue         # 這筆紀錄已過時（有更短的路已更新），跳過

        for neighbor, weight in graph[node]:
            new_dist = d + weight
            if new_dist < dist[neighbor]:   # 找到更短的路，更新
                dist[neighbor] = new_dist
                heapq.heappush(heap, (new_dist, neighbor))

    return dist  # {'A': 0, 'B': 2, 'C': 4, 'D': 5, 'E': 9}
```

時間複雜度：$O((V + E) \log V)$，使用優先佇列。

:::warning
Dijkstra 不適用於含有負權重邊的圖。如果邊的權重可以是負數，要改用 Bellman-Ford 演算法（$O(VE)$），它不是貪婪，而是反覆對所有邊做鬆弛（relaxation）。
:::

### 有向圖 vs 無向圖路徑

Dijkstra 同時適用有向和無向圖，差別在於建圖方式：

- **有向圖**：A → B 的邊只能從 A 走到 B
- **無向圖**：A - B 的邊在建圖時要加兩個方向（A→B 和 B→A）

```python
# 有向圖：A → B，只能單向走
graph['A'].append(('B', weight))

# 無向圖：A - B，雙向都能走，所以加兩條
graph['A'].append(('B', weight))
graph['B'].append(('A', weight))
```

### 樹上路徑

樹是無環的連通圖，任意兩點間路徑唯一，所以不需要 Dijkstra——直接 DFS 或 BFS 從起點找到終點，沿途記錄路徑即可。

如果是加權樹，路徑長度就是沿途邊的權重加總：

```python
from collections import deque

# tree = {'A': [('B', 3), ('C', 5)], 'B': [('A', 3), ('D', 2)], ...}
def tree_distance(tree, start, end):
    queue = deque([(start, 0)])  # (目前節點, 累計距離)
    visited = {start}

    while queue:
        node, dist = queue.popleft()
        if node == end:
            return dist          # 找到終點，回傳距離
        for neighbor, weight in tree[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append((neighbor, dist + weight))
```

## 經典範例：活動選擇問題

這是貪婪演算法最經典的應用之一，也很貼近現實：有一間會議室，一堆會議都想用它，每場會議有開始時間和結束時間。你只能選不衝突的會議，怎麼選才能讓場數最多？

假設有以下幾場會議：

| 會議 | 開始 | 結束 |
| :--- | :---: | :---: |
| A | 1 | 4 |
| B | 3 | 5 |
| C | 0 | 6 |
| D | 5 | 7 |
| E | 3 | 9 |
| F | 6 | 10 |
| G | 8 | 11 |

貪婪策略：**按結束時間由早到晚排序，每次選最早結束、且不與已選會議衝突的那場。**

邏輯直覺很清楚：結束越早，留給後面的時間就越多，能塞進去的會議就越多。

選擇過程：選 A → B 衝突跳過 → 選 D → C 衝突跳過 → F 衝突跳過 → 選 G。

最終選了 A、D、G，共 3 場。

時間複雜度：$O(n \log n)$，排序主導。

## 貪婪的常見陷阱

### 0/1 背包問題

有一個容量有限的背包，每件物品只能拿一次，怎麼裝才能讓總價值最大？

| 物品 | 重量 | 價值 | 性價比 |
| :--- | :---: | :---: | :---: |
| A | 1 | 6 | 6.0 |
| B | 3 | 10 | 3.3 |
| C | 4 | 12 | 3.0 |

背包容量 5。

- 貪婪（按性價比）：選 A + B，總價值 **16**
- 最優解：選 A + C，總價值 **18**

貪婪給了錯誤答案，需要動態規劃。

:::tip
有趣的是，如果物品可以切割（分數背包問題），貪婪就是對的。整數背包（每件只能完整拿或不拿）不行，因為無法保證貪婪選擇性質成立。
:::

## 貪婪 vs 動態規劃

| | 貪婪 | 動態規劃 |
| :--- | :--- | :--- |
| 決策方式 | 每步選局部最優，不回頭 | 考慮所有可能，記錄子問題答案 |
| 適用條件 | 需額外滿足貪婪選擇性質 | 只需最優子結構 |
| 時間複雜度 | 通常更低 | 通常較高 |
| 正確性保證 | 不一定，需要證明 | 只要狀態設計正確，保證正確 |
| 典型例子 | 活動選擇、Prim、Kruskal、Dijkstra | 背包問題、最長公共子序列 |

## 什麼時候考慮貪婪

遇到最佳化問題時，可以用這個流程判斷：

1. **找看看有沒有明顯的排序策略**：活動選擇按結束時間、Kruskal 按邊的權重、Dijkstra 按距離——直覺的排序往往是貪婪策略的起點。
2. **試著證明這個策略是對的**：最常用的手法是「交換論證」——假設最優解沒有採用貪婪選擇，把它換成貪婪選擇，結果不會變差，因此貪婪不影響最優性。
3. **如果證不了，找反例**：一旦找到一個讓貪婪失效的輸入，就換動態規劃。

貪婪演算法的魅力在於它通常簡單、快速，實作也直覺。難的不是寫程式，而是判斷它在你的問題上是否成立。
